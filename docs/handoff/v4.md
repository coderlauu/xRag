# v4 Handoff

**日期：** 2026-04-07  
**版本：** `v4 / Phase 2A`  
**状态：** active

---

## 1. 当前版本一句话目标

在统一导入、检索与可观测性基线上，引入可引用的 AI 问答与混合检索，让资料从可找回走向可直接复用。

---

## 2. 本轮边界

### 必须实现

- 基于个人知识库的 `AI 问答`
- `keyword + semantic` 混合检索
- 问答范围控制
- 答案证据链与无证据拒答
- 导入内容到向量索引的 freshness、诊断与回补基线
- 对应的交互、评估、运维与发布策略

### 明确不做

- 开放互联网联网回答
- 团队协作 / 权限模型
- 浏览器插件
- 移动端
- 全自动 agent 工作流
- 更大范围的动态页面抓取和反反爬能力

### 原型 / 文档中存在但本轮只做预留

- 单文档摘要、多文档对比摘要
- 相关内容推荐
- 问答历史与保存会话
- 搜索高级过滤器增强
- 多模型路由与更细粒度的模型策略

- 先总结 `Phase 2A` 的真实产品边界与信任边界
- 再定义问题输入、范围控制、答案卡片、引用与拒答交互

- 等产品、交互和评估边界稳定后，再单独进入技术方案设计节点

---

## 7. 进入技术方案前必须回答的问题

1. 如何在不破坏现有关键词搜索与导入主链路的前提下，引入向量索引与混合检索
2. AI 问答如何保证“每条结论都有引用”，并在证据不足时稳定拒答
3. 搜索结果范围、标签、来源类型、时间范围与单文档上下文，如何作为统一的问答 scope contract
4. 如何建立可重复的检索 / 问答评估集，以及延迟、成本、灰度与回滚基线

---

## 8. 已知风险 / 历史决策

- `v1` 到 `v3` 已完成统一导入、检索、OCR、链接抓取、时间线与运维基线，`v4` 必须增量演进，不重写主链路
- 若没有引用与拒答机制，AI 问答会直接破坏产品信任
- 向量索引 freshness、回补和失败诊断会直接影响“刚导入内容为何问不到”的用户体验
- 模型提供商、成本、延迟与隐私边界需要在规划阶段先定义问题清单，但当前还不提前冻结工程实现抽象
