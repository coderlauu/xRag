# xRag Phase 1B PRD

**日期：** 2026-04-02  
**版本：** `v2 / Phase 1B`  
**状态：** completed

---

## 1. 一句话定位

把 `Phase 1A` 已可用的知识收件箱升级为更可靠的文件导入系统，重点解决真实文件解析、大文件上传和生产稳态能力。

---

## 2. 背景

`Phase 1A` 已经证明最小闭环成立：

- 用户可以导入文本和文件
- 内容可以被搜索找回
- production 已可访问

下一阶段的核心矛盾不再是“有没有主链路”，而是：

- 文件导入能力还不够强
- 大文件上传还不够稳
- 生产问题可见性不足
- 发布后的回滚和诊断能力还不够清晰

---

## 3. 本期目标

- 支持 `pdf` 真实解析闭环
- 支持 multipart 大文件上传
- 在上传和解析失败时提供更明确的诊断信息
- 建立 production 观测和回滚的最小可执行能力

---

## 4. 非目标

- AI 问答
- 语义搜索
- OCR
- 浏览器插件
- 多租户与团队协作
- 移动端

---

## 5. 核心场景

### 场景 1：上传真实 PDF

用户上传 PDF，希望系统能提取正文并纳入搜索，而不是只返回模拟成功/失败。

### 场景 2：上传较大文件

用户上传较大的文档时，希望上传更稳定，并具备断点恢复或至少分段能力。

### 场景 3：解析失败后定位原因

用户或开发者需要知道失败是上传问题、文件问题、解析问题还是对象存储问题。

### 场景 4：生产问题定位与回滚

系统上线后，出现异常时希望能快速看到服务状态、错误来源，并具备最小回滚路径。

---

## 6. 本期范围

### P0 必做

1. 文本型 `pdf` 真实解析
2. multipart 上传链路
3. 上传完成与对象完整性校验增强
4. 解析错误分类与错误信息可见化
5. production 观测最小集
   - 结构化日志
   - 关键健康检查
   - 错误收敛入口
6. 部署回滚最小基线

### P1 预留

1. 扫描版 PDF OCR
2. 链接正文抓取
3. 搜索排序增强
4. 更细粒度的任务看板

---

## 7. 验收标准

1. 用户上传文本型 PDF 后，系统可进入真实解析并产出可搜索内容。
2. 大文件上传不再依赖单对象 PUT，可通过 multipart 完成。
3. 上传失败、解析失败、对象缺失等问题在系统内有可见错误信息。
4. production 服务异常时，可通过日志和健康检查快速定位。
5. 部署后至少具备一条明确回滚路径。

---

## 8. 对技术方案的要求

进入实现前，必须补齐：

- `Phase 1B` 的增量架构说明
- 数据模型变更
- API 变更
- worker 解析策略
- 观测与回滚方案
