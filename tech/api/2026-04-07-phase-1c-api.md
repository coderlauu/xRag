## 1. 设计目标

在 `Phase 1B` API 基线之上，补齐：

- 链接导入接口
- OCR 相关状态读取
- 文档处理时间线接口
- 搜索匹配解释与排序增强字段

基础前缀继续保持：

```text
/api/v1
```

---

## 2. 设计原则

1. 优先做增量字段和增量资源，不随意推翻 `Phase 1B` 路径
2. 文件导入与链接导入最终都进入 `documents`
3. 匹配解释由后端生成，前端直接消费
4. 诊断码稳定英文枚举，页面展示文案中文化

---

## 3. 文档接口增量

### 3.1 文档列表 / 搜索

`GET /api/v1/documents`

新增查询参数：

| 参数 | 说明 |
| --- | --- |
| `source_type` | `pdf/link/text/file` |
| `ocr_status` | `not_required/queued/processing/success/failed` |
| `diagnosis_code` | 稳定失败分类码 |

响应新增字段：

```json
{
  "items": [
    {
      "id": "doc_123",
      "title": "供应商扫描合同",
      "source_type": "pdf",
      "parse_status": "success",
      "ocr_status": "success",
      "upload_status": "uploaded",
      "diagnosis_code": null,
      "match_explanation": "标题命中“合同”，正文第 3 页 OCR 文本命中“违约金”。",
      "ranking_hint": "标题命中 + OCR 正文命中 + 最近导入加权"
    }
  ]
}
```

### 3.2 文档详情

`GET /api/v1/documents/{documentId}`

新增字段：

- `source_url`
- `ocr_status`
- `ocr_engine`
- `match_explanation`
- `ranking_hint`
- `matched_fields`

### 3.3 文档时间线

`GET /api/v1/documents/{documentId}/timeline`

响应示例：

```json
{
  "document_id": "doc_123",
  "items": [
    {
      "event_type": "ocr_started",
      "stage": "ocr",
      "status": "processing",
      "diagnosis_code": null,
      "summary": "OCR worker 开始处理第 1 批页面。",
      "created_at": "2026-04-07T08:20:00Z"
    }
  ]
}
```

### 3.4 重试动作

保留：

- `POST /api/v1/documents/{documentId}/retry`

当前约定：

- 仍统一走 `POST /api/v1/documents/{documentId}/retry`
- 当 `source_type=link` 时，后端自动创建 `fetch_link` 重试任务
- 当 `source_type!=link` 时，后端继续创建 `reparse_document` 重试任务

## 4. 链接导入接口

### 4.1 创建链接文档

`POST /api/v1/documents/link`

请求体：

```json
{
  "title": "可选标题覆盖",
  "source_url": "https://news.example.com/roadmap",
  "tags": ["路线图", "官方站点"]
}
```

行为：

- 校验 URL
- 创建 document
- 写入 `parse_status=pending`
- 入队 `fetch_link`

成功响应：

```json
{
  "id": "doc_789",
  "parse_status": "pending",
  "diagnosis_code": null
}
```

失败响应示例：

```json
{
  "error_code": "link_invalid_url",
  "message": "链接格式无效或不在允许范围内。"
}
```

## 6. 诊断码约定

新增：

- `ocr_runtime_error`
- `ocr_timeout`
- `ocr_no_text_detected`
- `link_fetch_timeout`
- `link_fetch_blocked`
- `link_extract_empty`
- `link_invalid_url`
- `search_projection_stale`

---

## 7. API 容易忽略的点

- 链接导入成功创建 document 不代表正文提取成功
- OCR 子状态和总 `parse_status` 必须同时返回，否则前端无法解释处理中与失败
- 时间线接口必须按时间倒序或正序固定，避免前端二次猜测
- 搜索解释字段不能只在详情页返回，搜索列表也要能直接消费

---

## 8. 测试要求

- 链接导入接口至少覆盖：成功、无效 URL、抓取失败
- OCR 相关接口至少覆盖：需要 OCR、OCR 成功、OCR 失败
- 文档时间线接口至少覆盖：文件导入成功链、链接抓取失败链
