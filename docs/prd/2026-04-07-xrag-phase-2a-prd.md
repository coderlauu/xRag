# xRag Phase 2A PRD

**日期：** 2026-04-07  
**版本：** `v4 / Phase 2A`  
**状态：** archived

---

## 1. 一句话定位

在已完成统一导入、检索与可观测性基线上，引入可引用的 AI 问答与混合检索，让资料从“可找回”升级到“可直接复用”。

---

## 2. 背景

`Phase 1A` 到 `Phase 1C` 已经连续验证了三件事：

- 内容可以被统一导入
- 内容可以被稳定搜索找回
- 处理过程、失败诊断与生产观测已有最小可用基线

当前新的核心矛盾不再是“资料进不来”或“资料搜不到”，而是：

- 用户仍然需要自己打开多篇文档、手动比对和总结
- 关键词搜索对“换一种说法提问”的支持仍然有限
- 即使搜索命中，用户也缺少一个可直接消费的答案层
- 如果直接引入 AI，但没有证据链与拒答机制，产品信任会明显下降

---

## 3. 本期目标

- 支持基于个人知识库的 `AI 问答`
- 支持 `keyword + semantic` 混合检索
- 支持基于搜索结果、标签、来源类型、时间范围、单篇文档的问答范围控制
- 提供可引用、可跳转、可审计的答案证据链
- 建立导入内容到向量索引的更新、诊断与回补基线
- 建立答案质量、检索质量、延迟与成本的最小评估框架

---

## 4. 非目标

- 开放互联网联网回答
- 无证据强答
- 团队协作 / 权限模型
- 浏览器插件
- 移动端
- 全自动 agent 工作流
- 更大范围的动态页面抓取和反反爬能力

---

## 5. 核心场景

### 场景 1：跨文档提问并获得带引用答案

用户输入一个自然语言问题，希望系统不仅返回搜索结果，还能直接给出答案，并标明答案来自哪些文档和片段。

### 场景 2：限定范围提问

用户已经筛出一组搜索结果，或只想问某个标签、来源类型、时间段内的内容，希望问答严格限定在该范围内。

### 场景 3：证据不足时拒答

用户提出的问题在知识库中证据不足时，系统应明确说明“当前证据不足”，而不是生成似是而非的内容。

### 场景 4：追溯答案来源

用户看到答案后，希望快速查看引用片段、跳到原始文档详情，并理解为什么这些证据被选中。

### 场景 5：新导入内容快速进入可问答状态

用户刚上传文件、OCR 文档或导入链接后，希望内容能在合理时间内进入 AI 问答索引；若失败，需要有诊断和修复入口。

---

## 6. 优先级原则

1. 先建立“可信问答闭环”，再扩展问题覆盖面和体验丰富度
2. 先解决“为什么能答 / 为什么不能答”，再优化“回答看起来多聪明”
3. 先覆盖用户最高频的问答入口，再补 power-user 范围筛选和运维面板
4. 先冻结评估门槛与 freshness 预期，再进入技术方案设计

---

## 7. 需求细化与优先级拆分

### 7.1 P0 首批交付

`P0` 的目标不是“把所有 AI 功能都放上来”，而是先做出可用且可信的首个闭环。

| ID | 能力 | 需求细化 | 为什么是 P0 |
| --- | --- | --- | --- |
| `P0-01` | 问答入口 | 提供独立的问答工作台，支持用户直接输入自然语言问题，并清晰展示 `retrieving / answered / needs_scope / refused` 状态。 | 没有一级问答入口，就无法验证“从搜索到答案”的产品升级是否成立。 |
| `P0-02` | 混合检索基线 | 问答必须建立在 `keyword + semantic` 混合检索之上，至少要比纯关键词更稳定地覆盖语义改写问题。 | 没有混合检索，AI 问答只是在弱召回上包一层生成。 |
| `P0-03` | 最小范围控制 | 首批范围控制至少支持 `全库 / 当前搜索结果 / 单篇文档`，并把当前 scope 作为正式产品事实展示给用户。 | 范围失控会直接破坏答案可信度；这三种已覆盖最高频场景。 |
| `P0-04` | 证据链 | 每个答案必须展示引用文档、引用片段和跳回详情页的能力；没有引用时不允许当作正常回答展示。 | 这是整个 `v4` 的信任底线。 |
| `P0-05` | 拒答与补救 | 当证据不足或范围过大时，系统必须显式返回 `refused` 或 `needs_scope`，并给出下一步动作。 | 没有这层，产品会默认用弱答案掩盖不确定性。 |
| `P0-06` | Freshness 可见性 | 新导入内容必须明确展示 `ready / stale / failed` 等问答就绪状态，并给出最小重试/回补入口。 | 用户最容易质疑“为什么刚导入却问不到”。 |
| `P0-07` | 评估门槛 | 必须有最小 golden set、质量门槛和上线前检查项，覆盖 retrieval、citation、refusal、latency、cost。 | 没有评估门槛，就无法判断产品是在变好还是变得更会胡说。 |

### 7.2 P1 补强

这些能力重要，但不是首个可信闭环的阻塞项，应在 `P0` 稳定后补上。

| ID | 能力 | 需求细化 | 为什么下放到 P1 |
| --- | --- | --- | --- |
| `P1-01` | 扩展范围控制 | 把 scope 扩展到 `标签 / 来源类型 / 时间范围`。 | 这会明显提升 power-user 价值，但不是验证可信问答闭环的首个门槛。 |
| `P1-02` | 检索实验台增强 | 展示 lexical / semantic / rerank 的贡献、未入答原因和 URL state。 | 更偏产品排查和内部调优，不是首个用户价值入口。 |
| `P1-03` | 评估与运维面板增强 | 在 `ops` 板上系统展示 recall、groundedness、citation coverage、latency、cost、backlog 等趋势。 | 指标本身是 P0，但完整面板形态可后补。 |
| `P1-04` | 证据包增强 | 支持更细粒度的 claim grouping、多引用组合、freshness 提示强化。 | 首版只要保证“有引用且可验证”即可，不必一步做到最细。 |
| `P1-05` | 问题历史 | 保留最近问题、最近答案与继续追问入口。 | 体验增强项，对首轮价值验证不是硬门槛。 |

### 7.3 明确延后

以下能力保留到 `Phase 2B+` 或后续版本，不放进当前阶段范围：

| 能力 | 原因 |
| --- | --- |
| 单文档摘要、多文档对比摘要 | 它们建立在可信问答闭环之上，当前不应抢占主线。 |
| 相关内容推荐 | 属于消费增强，不是首个问题闭环。 |
| 搜索高级过滤器增强 | 偏检索体验增强，不是 `v4` 的主价值跃迁。 |
| 多模型路由与更细粒度模型策略 | 当前先验证产品边界和评估门槛，不提前扩张模型层复杂度。 |
| 开放互联网联网回答、浏览器插件、团队协作、移动端 | 与当前单用户、私有知识库的可信问答主线无关。 |

## 8. 阶段切片建议

### Slice A: 可信问答 Alpha

先交付：

- `P0-01` 问答入口
- `P0-02` 混合检索基线
- `P0-03` 最小范围控制
- `P0-04` 证据链
- `P0-05` 拒答与补救

目标：

- 证明用户已经能“直接问”，且系统能够“有证据地答”或“有理由地拒答”。

### Slice B: Freshness 与评估闭环

继续补齐：

- `P0-06` Freshness 可见性
- `P0-07` 评估门槛
- `P1-02` 检索实验台增强
- `P1-03` 评估与运维面板增强

目标：

- 证明这条问答链路不仅能演示，还能被稳定验证、诊断和上线守护。

### Slice C: 范围与体验补强

最后再补：

- `P1-01` 扩展范围控制
- `P1-04` 证据包增强
- `P1-05` 问题历史

目标：

- 在不破坏信任边界的前提下，扩大可用场景和产品完成度。

## 9. 验收标准

### 9.1 P0 Gate

1. 用户可输入自然语言问题，并获得 `answered / needs_scope / refused` 之一的明确结果。
2. 每个 `answered` 结果都必须附带明确引用，用户可跳转到对应文档详情页验证来源。
3. 语义改写后的问题，相比纯关键词基线，召回质量有可感知提升。
4. 当知识库证据不足或范围过大时，系统能稳定拒答或提示缩小范围，而不是无依据回答。
5. 用户至少可以把问答范围限定到 `全库 / 当前搜索结果 / 单篇文档`。
6. 新导入的文本、PDF、OCR、链接内容能够明确展示问答索引 readiness；失败时可见诊断和回补入口。
7. 至少形成一套可重复执行的评估数据集与质量基线，用于衡量检索质量、答案 groundedness、延迟和成本。

### 9.2 Phase 2A 完整度补强

1. 范围控制可进一步扩展到标签、来源类型和时间范围。
2. 产品内可查看 retrieval trace、未入答原因和运维指标趋势。
3. 最近问题、证据包增强等体验项不会破坏 `P0 Gate` 的信任边界。

---

## 10. 对技术方案的要求

进入实现前，必须补齐：

- `Phase 2A` 的交互说明与原型
- 混合检索与向量索引架构
- chunk / embedding / retrieval / rerank 的最小策略
- 问答会话、答案引用、检索轨迹与索引任务的数据模型
- 问答 API、引用 contract 与拒答 contract
- 导入到索引的异步状态流与诊断方案
- 质量评估、灰度发布、feature flag 与回滚方案
