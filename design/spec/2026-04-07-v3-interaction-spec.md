## 1. 文档目的

本说明定义 `Phase 1C` 的页面结构、交互规则、状态流转和异常反馈，用于：

- 衔接 `Phase 1C PRD` 与后续技术方案
- 明确 `OCR / 链接抓取 / 搜索解释 / 时间线` 的页面行为
- 让前端、后端、worker 和运维共享同一份交互事实源

---

## 2. 页面总览

### 页面列表

- `index.html`：导入工作台
- `search.html`：搜索增强页
- `detail.html`：文档时间线页
- `ops.html`：运维看板

### 全局原则

- 页面重点从“是否成功”转向“为什么这样处理、为什么这样排序”
- 扫描件和链接都必须进入统一文档主链路，不允许形成隐形旁路
- 失败态必须展示建议动作，而不是只有错误码
- 时间线既服务用户排查，也服务运维定位

## 4. 导入工作台

### 页面目标

让用户理解 `Phase 1C` 已从“上传文件”扩展为“文件与链接统一导入”，并能清楚看到 OCR、抓取、投影和诊断的处理过程。

### 主要模块

1. 版本 Hero
2. 场景切换
3. 当前处理会话
4. 失败诊断面板
5. OCR / 抓取 / 搜索信号
6. 最近导入

### 核心交互

#### 4.1 场景切换

用户可切换预设场景：

- 扫描版 PDF OCR 成功
- 扫描版 PDF OCR 处理中
- 链接抓取成功
- 链接抓取失败

切换后同步更新：

- 中央时间线
- 右侧诊断说明
- 最近导入列表高亮

#### 4.2 会话展示

对于文件导入，时间线至少展示：

- `发起上传`
- `完成上传`
- `判断是否需要 OCR`
- `执行 OCR`
- `写入搜索投影`

对于链接导入，时间线至少展示：

- `提交链接`
- `抓取 HTML`
- `提取正文`
- `写入搜索投影`

#### 4.3 失败补救动作

失败态至少提供：

- 重新抓取
- 改为文本导入
- 查看文档时间线
- 打开运维看板

## 5. 搜索增强页

### 页面目标

让用户看到“为什么命中”和“为什么排序在这里”，而不是只看到结果列表。

### 主要模块

1. 关键词输入
2. 来源类型筛选
3. 解析状态筛选
4. 诊断类型筛选
5. 结果列表
6. 匹配与排查建议

### 核心交互

#### 5.1 结果解释

每个结果应显示：

- 命中来自标题、正文、OCR 文本还是链接正文
- 关键排序信号
- 如果失败，为何尚未进入可搜索状态

#### 5.2 失败结果

失败结果仍应出现在搜索里，便于用户理解：

- 卡在哪一步
- 推荐动作是什么
- 是否需要进入详情页或运维页

#### 5.3 URL State

关键词、来源类型、解析状态、诊断类型都应进入 URL，后续正式实现直接复用。

## 6. 文档时间线页

### 页面目标

为单条文档提供处理证据，展示：

- 接入模式
- 处理时间线
- 提取结果
- 匹配解释
- 失败诊断
- 关联事件

### 核心交互

#### 6.1 成功态

成功态需要展示：

- 提取正文
- 命中解释
- 页数或来源链接
- 时间线中的关键阶段

#### 6.2 失败态

失败态需要展示：

- 稳定错误码
- 中文摘要
- 推荐补救动作
- 是否存在关联事件

#### 6.3 时间线颗粒度

`Phase 1C` 的时间线必须比 `Phase 1B` 更细，至少能分清：

- 上传阶段
- 解析或抓取阶段
- OCR 阶段
- 搜索投影阶段

## 7. 运维看板

### 页面目标

让开发者和运维快速判断问题位于：

- API
- Worker
- OCR runtime
- 链接抓取器
- 搜索投影

### 主要模块

1. 服务健康
2. 降级与回滚基线
3. 最近 OCR / 抓取 / CI 事件
4. 推荐动作

### 核心交互

#### 7.1 运行时分层

必须显式展示 `OCR runtime` 和 `抓取器`，不能再只看 API / Worker 两层。

#### 7.2 回滚策略

应先考虑功能降级：

- 关闭 OCR feature flag
- 暂停链接抓取

只有整体不稳时，才建议回滚整套版本。

---

## 8. 容易忽略的交互点

- OCR 处理中时，搜索页应能看到“处理中”而不是直接消失
- 链接抓取失败时，详情页要告诉用户是否可以改为复制正文
- 匹配解释要避免术语堆积，排序原因需要可读
- 运维页不能泄露过多内部实现细节给普通用户，正式实现应有访问控制
