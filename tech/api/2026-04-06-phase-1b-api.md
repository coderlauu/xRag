## 1. 设计目标

在 `Phase 1A` API 基线之上，补齐：

- multipart 上传会话接口
- 诊断信息接口
- 详情页和运维页所需的增量读取
- 更稳定的失败语义

API 基础前缀继续保持：

```text
/api/v1
```

---

## 2. 设计原则

1. 尽量保持 `Phase 1A` 路径稳定，优先做增量字段和增量资源
2. 上传与解析继续解耦
3. 错误码稳定、中文文案可变
4. 列表接口可直接支撑页面筛选，不让前端二次拼装

## 4. 上传接口设计

## 4.1 初始化上传

`POST /api/v1/uploads/initiate`

请求体：

```json
{
  "file_name": "quarterly-review-q1.pdf",
  "mime_type": "application/pdf",
  "file_size": 18874368,
  "checksum_sha256": "optional-client-checksum"
}
```

服务端行为：

- 判断 `single` 还是 `multipart`
- 创建 upload session
- 返回单对象上传地址或 multipart 分片参数

响应示例：

```json
{
  "upload_id": "upl_123",
  "upload_mode": "multipart",
  "object_key": "uploads/2026/04/06/upl_123/quarterly-review-q1.pdf",
  "status": "initiated",
  "part_size_bytes": 5242880,
  "part_count": 4,
  "provider_upload_id": "s3-upload-id",
  "expires_in": 900
}
```

### 4.2 获取 part 上传地址

`POST /api/v1/uploads/{uploadId}/parts`

请求体：

```json
{
  "part_numbers": [1, 2, 3]
}
```

响应：

```json
{
  "upload_id": "upl_123",
  "parts": [
    {
      "part_number": 1,
      "upload_url": "https://storage.example.com/...",
      "headers": {}
    }
  ]
}
```

说明：

- 第一版也可按单个 `part_number` 获取签名，但批量接口更利于前端并发上传

### 4.3 完成 multipart part 回执

`POST /api/v1/uploads/{uploadId}/parts/{partNumber}/complete`

请求体：

```json
{
  "etag": "\"etag-from-storage\"",
  "size_bytes": 5242880
}
```

行为：

- 记录该 part 已成功
- 更新 `uploaded_part_count`

### 4.4 完成上传

`POST /api/v1/uploads/{uploadId}/complete`

请求体：

```json
{
  "title": "季度复盘 PDF 摘要",
  "tags": ["复盘", "pdf"],
  "checksum_sha256": "abc123",
  "parts": [
    { "part_number": 1, "etag": "\"etag-1\"" },
    { "part_number": 2, "etag": "\"etag-2\"" }
  ]
}
```

行为：

- 对 single upload：
  - 校验对象存在
- 对 multipart：
  - 校验 part 列表完整
  - 调用 provider complete
  - 校验对象存在和大小
- 创建 document
- 写入 `upload_status=uploaded`
- 写入 `parse_status=pending`
- 入队 parse job

成功响应：

```json
{
  "upload_id": "upl_123",
  "document_id": "doc_123",
  "job_id": "job_456",
  "upload_status": "uploaded",
  "parse_status": "pending",
  "diagnosis_code": null
}
```

失败响应示例：

```json
{
  "code": "object_missing_on_complete",
  "message": "上传完成后未在对象存储中找到目标文件。",
  "retryable": true
}
```

## 6. 运维接口

`Phase 1B` 需要最小可用的只读运维接口，建议统一放在 `/ops` 或 `/internal/ops` 下。

### 6.1 健康摘要

`GET /api/v1/ops/health-summary`

响应：

```json
{
  "services": [
    { "name": "api", "status": "healthy", "detail": "ready" },
    { "name": "worker", "status": "warning", "detail": "queue backlog 3" },
    { "name": "storage", "status": "healthy", "detail": "reachable" }
  ],
  "generated_at": "2026-04-06T03:00:00Z"
}
```

### 6.2 最近 incidents

`GET /api/v1/ops/incidents`

如果第一版不落库，可由 API 读取 GitHub issue / artifact 索引或本地聚合缓存。

响应：

```json
{
  "items": [
    {
      "incident_ref": "CI-240404-01",
      "source": "ci",
      "severity": "medium",
      "status": "resolved",
      "title": "CI production smoke timeout",
      "summary": "......",
      "external_url": "https://github.com/..."
    }
  ]
}
```

### 6.3 回滚摘要

`GET /api/v1/ops/deployments/latest`

返回：

- 当前镜像 tag
- 上一个稳定 tag
- 最近 smoke 结果

## 7. 错误语义

统一错误响应：

```json
{
  "code": "multipart_part_failed",
  "message": "第 17 个分片上传失败，请重试该分片。",
  "retryable": true,
  "context": {
    "upload_id": "upl_123",
    "part_number": 17
  }
}
```

规则：

- `code` 稳定英文
- `message` 可中文
- `retryable` 明确给前端补救建议

## 8. 测试要求

### 8.1 API 集成测试

- `initiate` 返回 single / multipart 两种模式
- part complete 更新上传进度
- complete 成功创建文档与任务
- object missing 返回稳定错误码

### 8.2 E2E

- PDF 成功导入
- multipart 中断恢复
- 失败文档进入搜索和详情页

### 8.3 运维读取

- 健康摘要接口可用
- incidents 接口可返回最近事件

---

## 9. 容易忽略的 API 问题

- multipart 请求体不要让前端上传过多无关字段
- complete 接口要能识别“客户端重复提交”
- 运维接口不能默认公开，应有访问控制
- presigned URL 与 public host 不一致时，错误码要能直达页面
