## 1. 文档目的

本说明定义 `Phase 1B` 的页面结构、交互规则、状态流转和异常反馈，用于：

- 衔接 `Phase 1B PRD` 与后续技术方案
- 明确 `pdf / multipart / observability` 的页面行为
- 给前端、后端、worker 和运维同一份交互事实源

---

## 2. 页面总览

### 页面列表

- `index.html`：导入页 / 导入工作台
- `search.html`：搜索页 / 搜索诊断
- `detail.html`：详情页 / 文档控制台
- `ops.html`：运维页 / 运维看板

### 全局原则

- 原型以 `状态流` 为核心，而不是单一表单页
- 每个关键步骤都必须有可见反馈、下一步动作和失败解释
- 同一个文档在不同页面的状态表达保持一致

## 3. 核心对象与状态

### 文档对象

关键字段：

- `id`
- `title`
- `source_type`
- `file_name`
- `size_label`
- `upload_mode`
- `parse_status`
- `upload_status`
- `diagnosis_code`
- `diagnosis_summary`
- `content_preview`
- `content_clean`
- `incident_ref`

### 上传状态

- `draft`：仅选择了文件，尚未发起上传
- `initiated`：已申请上传会话
- `uploading`：正在上传分片或单对象
- `verifying`：客户端已提交完成，服务端正在校验对象
- `uploaded`：对象已确认存在
- `failed`：上传失败，需提示原因和补救动作

### 解析状态

- `pending`
- `processing`
- `success`
- `failed`

### 失败诊断分类

- `storage_presign_failed`
- `multipart_part_failed`
- `object_missing_on_complete`
- `pdf_parse_unsupported`
- `pdf_parse_timeout`
- `queue_backlog`

## 4. 导入页 / 导入工作台

### 页面目标

让用户理解 `Phase 1B` 的导入链路已经从“直接上传”变成“上传会话 + 完成校验 + 解析队列 + 诊断反馈”。

### 主要模块

1. 版本 Hero
2. 核心能力卡
3. 上传实验台
4. 当前会话时间线
5. 失败诊断面板
6. 最近导入与任务看板
7. 运维入口

### 核心交互

#### 4.1 选择导入场景

用户可切换预设场景：

- 文本型 PDF 成功
- 大文件 multipart 上传中
- 完成校验失败
- 解析失败并生成事件

切换后：

- 中央时间线更新
- 右侧诊断说明更新
- 最近导入与任务列表高亮对应文档

#### 4.2 上传会话展示

上传会话必须显式显示：

- `initiate`
- `part upload`
- `complete`
- `verify object`
- `enqueue parse`
- `parse result`

#### 4.3 失败补救动作

失败态至少提供：

- 重试上传
- 重试解析
- 查看诊断详情
- 打开运维页

### 异常状态

- presigned URL 申请失败
- multipart 某一分片失败
- complete 成功提交但对象缺失
- 解析任务排队过久

## 5. 搜索页 / 搜索诊断

### 页面目标

让用户不仅能搜索到成功文档，也能定位失败文档、处理中任务和问题类型。

### 主要模块

1. 关键词输入
2. 来源类型筛选
3. 解析状态筛选
4. 诊断类型筛选
5. 结果列表
6. 结果摘要面板

### 核心交互

#### 5.1 结果按可执行性分组

结果列表支持快速聚焦：

- `可搜索`
- `处理中`
- `失败待处理`

#### 5.2 失败结果展示规则

失败结果必须展示：

- 失败原因摘要
- 最近一次失败时间
- 推荐动作
- 跳转详情或运维页

#### 5.3 URL State 约束

关键词、状态、诊断类型都应可进入 URL，便于后续正式前端复用。

## 6. 详情页 / 文档控制台

### 页面目标

让用户确认单条文档的导入、解析和故障信息，并在这里执行补救动作。

### 主要模块

1. 文档头部状态摘要
2. 正文 / 提取结果
3. 上传会话信息
4. 解析时间线
5. 失败诊断区
6. 相关事件 / 排查手册

### 核心交互

#### 6.1 成功态

展示：

- 提取成功的正文
- 页数 / 大小 / 文件类型
- 上传模式（单对象 / 分片上传）

#### 6.2 失败态

展示：

- 失败分类
- 错误说明
- 建议动作
- 最近一次 job / 事件引用

#### 6.3 重试动作

支持：

- `重试解析`
- `重新发起上传`
- `查看事件`

## 7. 运维页 / 运维看板

### 页面目标

为开发者和运维提供单一入口，快速判断问题位于 `API / Worker / Storage / Queue / Deploy` 哪一层。

### 主要模块

1. 服务健康卡
2. 最近部署与回滚入口
3. 最近 CI 事件
4. 错误分布
5. 推荐排查动作

### 核心交互

#### 7.1 服务分层查看

至少可看到：

- API
- Worker
- PostgreSQL
- Redis
- 对象存储

#### 7.2 事件列表

当 CI 或导入链路失败时，应能从这里看到：

- 标题
- 触发时间
- 来源（部署 / 上传 / 解析）
- 状态

#### 7.3 回滚说明

运维页要明确展示：

- 当前生产镜像 tag
- 上一个稳定 tag
- 回滚步骤摘要

---

## 8. Prototype 特有行为

以下是 `v2` 原型中的演示逻辑，不代表正式实现：

- 所有状态切换由前端 mock 数据驱动
- 上传 / 解析 / 事件 为静态场景切换
- 服务健康和部署信息为演示数据

正式系统需要分别由：

- API + storage
- worker + queue
- 部署流水线 + 事件采集器

来承接。

## 9. 技术 Handoff 关注点

`Phase 1B` 技术方案必须回答：

1. multipart 会话的 API 边界如何设计
2. `upload_status` 与 `parse_status` 的组合状态如何存储

4. 运维页需要哪些可复用的读接口
5. 失败闭环如何与现有 `CI Failure Loop` 资产接起来
