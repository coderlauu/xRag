## 1. 本文档目的

这份文档用于冻结 `Phase 3A` 的 API 方向：

- 哪些现有 answer/document/ops 路径继续保留
- `diagnostic sample / replay / deployment compare` 应如何挂在 `ops` 资源族下
- 哪些响应应复用既有 DTO，哪些应形成新的 wrapper read model

本阶段不直接生成 OpenAPI，只先冻结资源方向和响应分层。

---

## 2. 当前 API 事实

当前已存在的相关只读资源：

- `GET /api/v1/answers`
- `GET /api/v1/answers/:sessionId`
- `GET /api/v1/answers/:sessionId/retrieval`
- `GET /api/v1/documents`
- `GET /api/v1/documents/:documentId`
- `GET /api/v1/documents/:documentId/evidence`
- `GET /api/v1/documents/:documentId/timeline`
- `GET /api/v1/ops/overview`
- `GET /api/v1/ops/trends`
- `GET /api/v1/ops/incidents`
- `GET /api/v1/ops/deployments/latest`

当前问题：

1. 已有事实面是分散存在的，缺少统一的诊断对象和入口上下文。
2. `ops` 主板能回答“哪里有风险”，但没有正式的样本列表和回放资源。
3. release guard 已有最新 deployment 摘要，但没有 compare 资源来回答“哪些样本变坏了”。

---

## 3. 资源方向与结论

### 3.1 现有资源继续保留

以下路径继续保留，不重命名、不改语义：

- `GET /api/v1/answers/:sessionId`
- `GET /api/v1/answers/:sessionId/retrieval`
- `GET /api/v1/documents/:documentId`
- `GET /api/v1/documents/:documentId/evidence`
- `GET /api/v1/documents/:documentId/timeline`
- `GET /api/v1/ops/overview`
- `GET /api/v1/ops/trends`
- `GET /api/v1/ops/incidents`
- `GET /api/v1/ops/deployments/latest`

原因：

- 这些路径已经是事实面或稳定兼容面。
- `Phase 3A` 需要的是在其上方增加诊断 read model，而不是推翻既有路径。

### 3.2 `Phase 3A` 推荐新增的 `ops` 资源

推荐新增四个只读资源：

1. `GET /api/v1/ops/samples`
2. `GET /api/v1/ops/replays/answer-sessions/:sessionId`
3. `GET /api/v1/ops/replays/documents/:documentId`
4. `GET /api/v1/ops/deployments/compare`

**推荐理由**

- `samples`
  - 负责把 trends / incident / release guard 变成具名样本列表
- `replays/*`
  - 负责把现有 answer/document 事实组织成诊断工作流
- `deployments/compare`
  - 负责围绕一个 deployment anchor 形成 before/after compare，而不是把 compare 继续塞进 `deployments/latest`

### 3.3 当前不新增的路径

`Phase 3A / P0` 不推荐新增：

- `POST /api/v1/ops/*`
- `POST /api/v1/evals/*`
- `POST /api/v1/deployments/*`
- `/api/v1/logs/*`

原因：

- 当前目标仍是只读诊断产品化，不是执行编排、评测管理或日志平台。

## 4. 推荐资源形状

### 4.1 `GET /api/v1/ops/samples`

#### 目标

返回从 trends / incident cluster / release compare 进入的统一样本列表。

#### 推荐查询参数

- `origin`
  - `trend | incident_cluster | release_compare`
- `sample_kind`
  - `answer_session | document_pipeline`
- `window`
  - `24h | 7d | 30d`
- `cluster_key`
  - 可选，仅 incident/release 入口使用
- `deployment_record_id`
  - 可选，仅 release compare 入口使用
- `page`
- `page_size`

#### 推荐响应层级

```json
{
  "generated_at": "2026-04-17T00:00:00Z",
  "origin": "incident_cluster",
  "items": []
}
```

#### 每个 item 至少包含

- `sample_id`
- `sample_kind`
- `source_id`
- `severity`
- `detected_at`
- `title`
- `summary`
- `related_incident_ref`
- `related_deployment_record_id`
- `next_replay_ref`

### 4.2 `GET /api/v1/ops/replays/answer-sessions/:sessionId`

#### 目标

返回单次问答的诊断回放对象。

#### 推荐响应层级

```json
{
  "generated_at": "2026-04-17T00:00:00Z",
  "sample": {},
  "session": {},
  "retrieval": {},
  "related_context": {}
}
```

#### 方向结论

- `session`
  - 复用现有 `AnswerSessionResponse`
- `retrieval`
  - 复用现有 `AnswerRetrievalTraceResponse`
- `related_context`
  - 新增 replay wrapper 字段，承载：
    - `related_incident_ref`
    - `related_deployment_record_id`
    - `related_evaluation_run_ref`
    - `freshness_flags`

这样可以避免重复定义 question、scope、citation、retrieval 基础字段。

### 4.3 `GET /api/v1/ops/replays/documents/:documentId`

#### 目标

返回单 document 的 pipeline replay。

#### 推荐响应层级

```json
{
  "generated_at": "2026-04-17T00:00:00Z",
  "sample": {},
  "document": {},
  "timeline": {},
  "evidence": {},
  "related_context": {}
}
```

#### 方向结论

- `document`
  - 复用现有 `DocumentDetail`
- `timeline`
  - 复用现有 `DocumentTimelineResponse`
- `evidence`
  - 复用现有 `DocumentEvidenceResponse`
- `related_context`
  - 新增：
    - `blocking_reason`
    - `related_incident_ref`
    - `related_answer_session_count`
    - `related_deployment_record_id`

### 4.4 `GET /api/v1/ops/deployments/compare`

#### 目标

围绕一个 deployment anchor 返回 before/after compare 与受影响样本。

#### 推荐查询参数

- `deployment_record_id`
  - 必填
- `window`
  - `24h | 7d | 30d`
- `sample_kind`
  - 可选，默认返回全部

#### 推荐响应层级

```json
{
  "generated_at": "2026-04-17T00:00:00Z",
  "deployment": {},
  "baseline": {},
  "before_window": {},
  "after_window": {},
  "delta_summary": {},
  "affected_samples": []
}
```

#### 关键约束

- `deployment`
  - 应以 `deployment_records` 为主事实源
- `baseline`
  - 可引用 `previous_stable_image_tag` 与最近相关 `evaluation_run`
- `affected_samples`
  - 直接返回 `diagnostic_sample` 列表，而不是新的 deployment-specific item

## 5. Shared Types 方向

后续 contract freeze 至少需要新增的 shared types：

- `OpsDiagnosticSample`
- `OpsDiagnosticSampleListResponse`
- `OpsDiagnosticSampleKind`
- `OpsDiagnosticOrigin`
- `OpsAnswerSessionReplayResponse`
- `OpsDocumentReplayResponse`
- `OpsDeploymentCompareResponse`

并继续复用现有 shared types：

- `AnswerSessionResponse`
- `AnswerRetrievalTraceResponse`
- `DocumentDetail`
- `DocumentTimelineResponse`
- `DocumentEvidenceResponse`
- `OpsOverviewResponse`
- `OpsTrendsResponse`

---

## 6. Freeze-Ready 结论

1. `Phase 3A` 的 API 增量应全部挂在 `ops` 资源族下。
2. 现有 `answers / documents` 路径继续保留为事实面，不直接重命名成 replay API。
3. replay 响应应优先通过 wrapper 复用既有 DTO，而不是平铺复制已有字段。
4. deployment compare 应以 `deployment_record_id` 为必填 anchor。
5. 当前没有新的 API blocker；下一步可以进入 contract freeze，锁定路径、query、枚举和值对象形状。
