## 1. 文档目的

这份文档把 `Phase 2A PRD` 中的 `P0 / P1 / deferred` 继续细化成可执行 backlog，用于：

- 约束后续技术方案只围绕真实主线展开
- 让产品、设计、工程和 agent 对“首批必须交付什么”有统一理解
- 为后续 coding handoff 准备逐项验收清单

当前阶段仍处于产品调研与需求冻结，不在这里定义正式实现方案。

---

## 2. 排序原则

1. 先交付“可信问答闭环”，再交付“更强功能覆盖”
2. 先保证用户知道“为什么能答 / 为什么不能答”，再优化炫技型能力
3. 任何新能力都不能破坏 `citation / refusal / freshness` 三条信任边界
4. `P1` 和 `deferred` 不得反向挤占 `P0` 的资源与设计空间

## 3. P0 Backlog

### `P0-01` 问答入口

**用户价值**

- 用户能直接把知识库当作“可提问系统”使用，而不是先搜再自己读。

**范围内**

- 一级导航中的独立问答工作台
- 自然语言问题输入
- 明确展示 `retrieving / answered / needs_scope / refused`
- 当前问题对应的 scope 摘要

**明确不含**

- 多轮对话
- 问题收藏、历史归档
- 联网问答

**验收清单**

- [ ] 用户能在一级入口直接发起提问
- [ ] 提问后页面不会停留在静态搜索状态，而会进入明确的问答状态机
- [ ] `answered / needs_scope / refused` 三类结果的展示差异清晰
- [ ] 当前问题的 scope 对用户可见，不是隐藏上下文

**前置依赖**

- `P0-02`
- `P0-03`
- `P0-04`
- `P0-05`

### `P0-02` 混合检索基线

**用户价值**

- 用户换一种说法提问时，仍能召回正确资料，不再完全依赖标题关键词。

**范围内**

- `keyword + semantic` 作为问答默认召回基线
- 至少能处理单事实和跨文档问题的基础召回
- 与现有搜索事实源保持一致

**明确不含**

- 多模型 rerank 策略竞争
- 复杂检索实验开关
- 召回策略 A/B 平台

**验收清单**

- [ ] 语义改写问题相比纯关键词有可感知提升
- [ ] 问答与搜索不会出现两套互相矛盾的结果来源
- [ ] 混合检索不会绕开当前文档主链路单独形成旁路

**前置依赖**

- `Phase 1C` 的统一导入与搜索基线

### `P0-03` 最小范围控制

**用户价值**

- 用户可以控制答案的证据边界，避免“全库乱答”。

**范围内**

- `全库`
- `当前搜索结果`
- `单篇文档`

**明确不含**

- 标签范围
- 来源类型范围
- 时间范围

**验收清单**

- [ ] 用户可在提问前或提问时看到当前 scope
- [ ] 切换 scope 后，旧答案不会被误当成新范围下的答案继续展示
- [ ] `当前搜索结果` 与 `单篇文档` 两类范围都可被稳定识别
- [ ] 当 scope 过大时，系统可以回到 `needs_scope` 而不是硬答

**前置依赖**

- `P0-01`

### `P0-04` 证据链

**用户价值**

- 用户不仅拿到答案，还能验证答案来自哪篇文档、哪段内容。

**范围内**

- 引用文档
- 引用片段
- 跳回详情页
- 明确展示答案引用数量

**明确不含**

- 复杂 claim tree
- 多引用聚类对比
- 高亮原文中的多层交叉标注

**验收清单**

- [ ] 每个 `answered` 结果都至少有一个明确 citation
- [ ] citation 至少能定位到文档与片段级别
- [ ] 用户可从答案跳到详情页做核查
- [ ] 没有 citation 的结果不能作为正常回答通过验收

**前置依赖**

- `P0-02`

### `P0-05` 拒答与补救

**用户价值**

- 当系统不能可靠回答时，用户能知道原因和下一步，而不是被弱答案误导。

**范围内**

- `refused`
- `needs_scope`
- 推荐动作文案

**明确不含**

- 自动补资料
- 自动扩 scope
- agent 自主继续追问

**验收清单**

- [ ] 证据不足时可以进入 `refused`
- [ ] 范围过大或冲突明显时可以进入 `needs_scope`
- [ ] `refused` 与 `needs_scope` 的原因和推荐动作不同
- [ ] 页面不会把失败态伪装成普通弱答案

**前置依赖**

- `P0-03`
- `P0-04`

### `P0-06` Freshness 可见性

**用户价值**

- 用户知道资料是否已经进入“可问答状态”，不会把索引延迟误判成答案质量差。

**范围内**

- `ready / stale / failed` 等就绪状态可见
- 最小重试或回补入口
- 刚导入内容为什么暂时问不到的解释

**明确不含**

- 完整索引治理台
- 大规模自动修复编排

**验收清单**

- [ ] 新导入文档的问答 readiness 对用户可见
- [ ] `stale` 与 `failed` 的含义可区分
- [ ] 当资料尚未 ready 时，问答页不会假设它已经可回答
- [ ] 用户至少能看到最小回补动作或诊断提示

**前置依赖**

- `Phase 1C` 的处理状态与诊断基线

### `P0-07` 评估门槛

**用户价值**

- 团队能判断这个版本是否真的可信，而不是只凭 demo 观感推进。

**范围内**

- 最小 golden set
- retrieval / answer / refusal / latency / cost 门槛
- 上线前检查项

**明确不含**

- 完整自动评测平台
- 长周期趋势看板
- provider 级预算中心

**验收清单**

- [ ] 已定义最小 golden set 规模与题型分布
- [ ] 已定义 `Recall@10 / groundedness / citation_coverage / refusal_precision / latency_p95 / avg_token_cost_usd` 等门槛
- [ ] 已明确“不达标时优先降 scope”的处理原则
- [ ] 技术方案设计前，产品侧已承认这些门槛是版本 gate

**前置依赖**

- [Phase 2A Evaluation Plan](/Users/coderlauu/xRag/docs/process/2026-04-07-phase-2a-evaluation-plan.md)

## 4. P1 Backlog

### `P1-01` 扩展范围控制

- 目标：把 scope 扩展到 `标签 / 来源类型 / 时间范围`
- 进入条件：`P0-03` 已稳定，且不会影响最小范围控制的可解释性
- 验收重点：扩展后的范围仍然能被用户明确感知，不引入隐藏过滤器

### `P1-02` 检索实验台增强

- 目标：完整展示 lexical / semantic / rerank 的贡献、未入答原因和 URL state
- 进入条件：`P0-02` 与 `P0-04` 已稳定
- 验收重点：这页服务解释和诊断，不改变主问答入口的语义

### `P1-03` 评估与运维面板增强

- 目标：在产品内可见 recall、groundedness、citation coverage、latency、cost、backlog 等趋势
- 进入条件：`P0-07` 的门槛已定
- 验收重点：指标定义必须与评估计划完全一致

### `P1-04` 证据包增强

- 目标：支持更细粒度的 claim grouping、多引用组合和更强 freshness 提示
- 进入条件：`P0-04` 已被验证足够稳定
- 验收重点：增强后的证据展示仍不能牺牲可读性

### `P1-05` 问题历史

- 目标：保留最近问题、最近答案与继续追问入口
- 进入条件：主问答闭环已完成
- 验收重点：历史能力不应改变 `P0` 的判定逻辑或状态语义

---

## 5. Deferred

- 单文档摘要、多文档对比摘要
- 相关内容推荐
- 搜索高级过滤器增强
- 多模型路由与更细粒度模型策略
- 开放互联网联网回答
- 浏览器插件
- 团队协作 / 权限模型
- 移动端

这些能力统一视为后续阶段候选项，不得在 `Phase 2A` 的技术方案设计中默认带入。

---

## 6. 阶段顺序

1. `Alpha`: `P0-01 ~ P0-05`
   目标：先验证“能提问、能答、能拒答、能引用”
2. `Beta`: `P0-06 ~ P0-07`
   目标：补齐 freshness 和评估 gate，让产品可被稳定验证
3. `Enhancement`: `P1-01 ~ P1-05`
   目标：扩展范围和解释能力，但不重写信任边界
