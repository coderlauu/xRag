## 1. API 设计原则

- 前后端分离，所有前端数据来自 HTTP API
- 以资源为中心，不暴露原型内部实现细节
- 长耗时解析不走同步阻塞接口
- 列表接口必须支持筛选与分页
- 状态字段与原型保持一致，避免前端再做映射猜测

基础前缀建议：

```text
/api/v1
```

## 2. 资源总览

| 资源 | 作用 |
| --- | --- |
| `/documents` | 文档查询、详情、标签更新、重试 |
| `/uploads` | 对象存储上传会话 |
| `/tags` | 标签查询与新增 |
| `/jobs` | 任务状态查询 |
| `/health` | 健康检查 |

## 3. 文档接口

### 3.1 新增文本文档

`POST /api/v1/documents/text`

请求体：

```json
{
  "title": "RAG 产品最小闭环拆解",
  "content": "这里是一段用户手动输入的正文",
  "tags": ["RAG", "MVP", "检索"]
}
```

行为：

- 直接创建 `documents`
- `parse_status=success`
- 同步生成 `content_preview`、`search_text`

响应：

```json
{
  "id": "doc_123",
  "parse_status": "success"
}
```

### 3.2 文档列表 / 搜索

`GET /api/v1/documents`

查询参数：

| 参数 | 说明 |
| --- | --- |
| `q` | 关键词 |
| `source_type` | `text/file/link` |
| `parse_status` | `pending/processing/success/failed` |
| `tags` | 逗号分隔标签名或 tag id |
| `date_from` | 开始时间 |
| `date_to` | 结束时间 |
| `page` | 页码 |
| `page_size` | 每页大小，默认 `20` |

响应示例：

```json
{
  "items": [
    {
      "id": "doc_123",
      "title": "RAG 产品最小闭环拆解",
      "content_preview": "……命中摘要……",
      "tags": ["RAG", "MVP", "检索"],
      "source_type": "file",
      "source_origin": "upload",
      "file_name": "rag-mvp.pdf",
      "parse_status": "success",
      "imported_at": "2026-03-31T12:00:00Z"
    }
  ],
  "page": 1,
  "page_size": 20,
  "total": 1
}
```

### 3.3 文档详情

`GET /api/v1/documents/{documentId}`

返回字段应覆盖原型详情页需要的：

- `title`
- `content_raw`
- `content_clean`
- `tags`
- `source_type`
- `source_origin`
- `source_url`
- `file_name`
- `mime_type`
- `parse_status`
- `parse_error_message`
- `created_at`
- `imported_at`

### 3.4 更新文档标签

`PATCH /api/v1/documents/{documentId}/tags`

请求体：

```json
{
  "tags": ["用户研究", "访谈", "P1"]
}
```

行为：

- 对不存在的标签先插入 `tags`
- 再重建 `document_tags`
- 刷新 `search_text`

### 3.5 重试解析

`POST /api/v1/documents/{documentId}/retry`

行为：

- 仅 `failed` 或 `success` 允许重试
- 重试后状态回到 `pending`
- 创建一条新的 `document_parse_jobs`

响应：

```json
{
  "document_id": "doc_123",
  "job_id": "job_456",
  "parse_status": "pending"
}
```

## 4. 上传接口

### 4.1 初始化上传

`POST /api/v1/uploads/initiate`

请求体：

```json
{
  "file_name": "rag-mvp.pdf",
  "mime_type": "application/pdf",
  "file_size": 10485760
}
```

响应示例：

```json
{
  "upload_id": "upl_123",
  "object_key": "uploads/2026/03/31/upl_123/rag-mvp.pdf",
  "upload_method": "presigned_put",
  "upload_url": "https://storage.example.com/...",
  "headers": {
    "content-type": "application/pdf"
  },
  "expires_in": 900
}
```

大文件时可返回 multipart upload 参数，前端分片上传。

### 4.2 完成上传并创建文档

`POST /api/v1/uploads/{uploadId}/complete`

请求体：

```json
{
  "title": "RAG 产品最小闭环拆解",
  "tags": ["RAG", "MVP"],
  "checksum_sha256": "abc123..."
}
```

行为：

- 校验上传会话存在且对象已上传
- 创建 `documents`
- `parse_status=pending`
- 入队 `parse_document`

响应：

```json
{
  "document_id": "doc_123",
  "job_id": "job_456",
  "parse_status": "pending"
}
```

## 5. 标签接口

### 5.1 获取标签列表

`GET /api/v1/tags`

查询参数：

- `status=active|archived`
- `q`

响应：

```json
{
  "items": [
    {
      "id": "tag_1",
      "name": "用户研究",
      "status": "active"
    }
  ]
}
```

### 5.2 新增标签

`POST /api/v1/tags`

请求体：

```json
{
  "name": "产品策略"
}
```

行为：

- 归一化
- 若存在同名归一化标签，返回已存在记录

## 6. 任务接口

### 6.1 查询任务状态

`GET /api/v1/jobs/{jobId}`

响应：

```json
{
  "id": "job_456",
  "document_id": "doc_123",
  "job_type": "parse_document",
  "status": "running",
  "attempt": 1,
  "error_message": null
}
```

说明：

- Inbox 页可轮询该接口
- 也可以直接轮询 `GET /documents/{id}`，以文档状态为准

Phase 1A 优先使用轮询，不强制引入 WebSocket / SSE。

## 7. 健康检查接口

### 7.1 基础健康检查

`GET /api/v1/health`

返回：

```json
{
  "status": "ok"
}
```

### 7.2 就绪检查

`GET /api/v1/health/ready`

检查：

- PostgreSQL 连通性
- Redis 连通性
- Object Storage 凭证可用性

## 8. 错误码约定

| 错误码 | 场景 |
| --- | --- |
| `UPLOAD_TYPE_UNSUPPORTED` | 不支持的文件类型 |
| `UPLOAD_TOO_LARGE` | 超出大小限制 |
| `UPLOAD_EXPIRED` | 上传会话已过期 |
| `DOCUMENT_NOT_FOUND` | 文档不存在 |
| `PARSE_TEXT_EXTRACTION_FAILED` | 文本抽取失败 |
| `PARSE_SCANNED_PDF_UNSUPPORTED` | 扫描版 PDF 当前不支持 |
| `TAG_NAME_INVALID` | 标签名非法 |

错误响应结构建议统一：

```json
{
  "code": "UPLOAD_TOO_LARGE",
  "message": "File exceeds maximum allowed size",
  "request_id": "req_123"
}
```

## 9. 前后端契约管理

### 9.1 推荐方式

- NestJS 生成 OpenAPI
- 前端通过 `orval` 或 `openapi-typescript` 生成 SDK
- CI 中校验 OpenAPI 是否发生破坏性变更

### 9.2 为什么重要

- 避免前端手写类型与后端漂移
- agent 与开发者都能直接读取机器可消费的接口事实

## 12. 结论

这套 API 设计的核心是：

- 用同步接口处理“创建请求”
- 用异步任务处理“文件解析”
- 用统一文档资源承接三页页面
- 用 OpenAPI 保证前后端分离后仍可稳定协作

它足以把原型中的三页闭环替换成真实系统，同时不把未来的 AI 能力强行混进当前版本。
