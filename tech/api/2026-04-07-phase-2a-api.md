## 1. 设计目标

`Phase 2A` 的 API 不追求一次性把所有问答能力都塞进来，而是围绕 `P0 可信问答闭环` 回答三个问题：

1. 如何在不推翻现有 `documents` 接口的前提下，让用户发起问题并读取结果
2. 如何让 citation、refusal、freshness 进入正式 contract
3. 如何保留当前 `Phase 1C` 的搜索、详情、时间线与 ops 主链路

基础前缀继续保持：

```text
/api/v1
```

---

## 2. API 设计原则

1. 不把 hybrid retrieval 粗暴塞回 `GET /documents`
2. 问答使用独立 `answers` 资源，而不是让前端自行拼装
3. `answered / needs_scope / refused / failed` 必须是正式状态，不使用模糊布尔值
4. 任何 citation 都必须能回到 `document + chunk` 级事实源
5. `P0` 的评估集管理先不做成运行时 API

## 3. 现有接口的增量改动

### 3.1 `GET /api/v1/documents`

继续作为：

- keyword search 基线
- `search_result` scope 的来源页面
- 列表页与当前详情页的数据来源

新增响应字段：

- `index_status`
- `indexed_at`
- `citation_ready`

可选新增查询参数：

- `index_status`

**不建议在 `P0` 新增**

- `retrieval_mode`
- 语义检索 explain
- answer-specific query 参数

理由：

- `P0` 的 hybrid retrieval 属于问答内部能力，不属于通用文档列表 API。
- 否则会把 `P1-02` 的检索实验台能力提前挤进主搜索接口。

### 3.2 `GET /api/v1/documents/{documentId}`

新增字段：

- `index_status`
- `index_version`
- `indexed_at`
- `citation_ready`

### 3.3 `GET /api/v1/documents/{documentId}/timeline`

沿用现有路径，但允许返回新增的 `stage=index` 事件。

### 3.4 新增 `GET /api/v1/documents/{documentId}/evidence`

用途：

- 详情页查看 chunk、citation locator 和索引 readiness

响应示例：

```json
{
  "document_id": "doc_123",
  "index_status": "ready",
  "citation_ready": true,
  "items": [
    {
      "chunk_id": "chk_001",
      "chunk_index": 0,
      "section_label": "Clause 3",
      "page_ref": "3",
      "quote_text": "第三条约定违约金比例为合同金额的 8%。",
      "locator": {
        "page": 3
      }
    }
  ]
}
```

### 3.5 新增 `POST /api/v1/documents/{documentId}/reindex`

用途：

- 触发重新 chunk / embed

说明：

- 不复用当前 `retry` 路径，避免把“解析重试”和“问答索引重建”混成一件事。

## 4. 新增问答资源

### 4.1 `POST /api/v1/answers`

用途：

- 创建 answer session
- 立即返回 session id 和初始状态

推荐请求体：

```json
{
  "question": "接下来两个月最值得投入的能力是什么？",
  "scope": {
    "mode": "search_result",
    "payload": {
      "query": "路线图 检索",
      "filters": {
        "source_type": "link"
      },
      "document_ids": ["doc_1", "doc_2", "doc_3"]
    }
  }
}
```

推荐响应：

```json
{
  "session_id": "ans_123",
  "status": "retrieving"
}
```

说明：

- `P0` 推荐 `202 Accepted` 语义。
- 前端通过轮询读取后续状态。

### 4.2 `GET /api/v1/answers/{sessionId}`

用途：

- 读取 answer session 当前状态和最终结果

响应示例：

```json
{
  "session_id": "ans_123",
  "question": "接下来两个月最值得投入的能力是什么？",
  "scope": {
    "mode": "search_result",
    "payload": {
      "document_ids": ["doc_1", "doc_2"]
    }
  },
  "status": "answered",
  "answer_summary": "基于当前知识库，优先投入混合检索、引用链路和 freshness。",
  "refusal_reason": null,
  "diagnosis_code": null,
  "retrieval_mode": "hybrid",
  "citations": [
    {
      "document_id": "doc_1",
      "chunk_id": "chk_01",
      "quote_text": "五月前优先投入混合检索、引用链路与索引 freshness。",
      "locator": {
        "anchor": "roadmap-retrieval"
      }
    }
  ],
  "latency_ms": 4210,
  "total_cost_usd": "0.0140"
}
```

### 4.3 `GET /api/v1/answers/{sessionId}/retrieval`

用途：

- 读取 retrieval trace
- 作为 `P1-02` 检索实验台的事实源

说明：

- 即使 `P1-02` 页面形态延后，`P0` 仍建议在 API 层保留该资源，便于排查和后续扩展。

响应示例：

```json
{
  "session_id": "ans_123",
  "items": [
    {
      "document_id": "doc_1",
      "chunk_id": "chk_01",
      "rank": 1,
      "lexical_score": 0.62,
      "semantic_score": 0.88,
      "final_score": 0.93,
      "used_in_answer": true,
      "exclusion_reason": null
    }
  ]
}
```

## 5. Scope Contract

### 5.1 `scope.mode`

`P0` 只允许：

- `global`
- `search_result`
- `document`

### 5.2 `scope.payload`

推荐约束：

- `global`
  - 空对象或 `null`
- `document`
  - 必须含 `document_id`
- `search_result`
  - 必须含 `document_ids`
  - 推荐同时携带 `query / filters`

### 5.3 为什么不在 `P0` 放更多模式

- `tag / source_type / time_range` 已明确排到 `P1`
- 若 API 先放开这些模式，会反向推动前端和数据层提前实现不在 `P0` 范围内的能力

## 6. Answer 状态与诊断码 contract

### 6.1 状态

- `retrieving`
- `synthesizing`
- `answered`
- `needs_scope`
- `refused`
- `failed`

### 6.2 推荐诊断码

- `retrieval_no_hits`
- `retrieval_scope_empty`
- `answer_insufficient_evidence`
- `citation_missing`
- `provider_timeout`

### 6.3 contract 规则

- `needs_scope` 必须给出用户可执行的缩小范围建议
- `refused` 必须明确说明证据不足
- `failed` 不得复用为“证据不足”

## 7. Ops 增量接口

### 7.1 新增 `GET /api/v1/ops/answer-summary`

至少包含：

- `embedding_backlog`
- `ready_document_count`
- `stale_document_count`
- `answer_latency_p95`
- `citation_coverage`
- `refusal_rate`
- `avg_token_cost_usd`

### 7.2 暂不新增

- `GET /api/v1/evals/cases`
- `GET /api/v1/evals/runs`

理由：

- `P0` 的评估集先保留为 repo 资产
- 运行时只需要聚合结果，不需要产品内管理评估样本

## 8. 与现有 shared types 的关系

`Phase 2A` 进入编码前，需要在 `packages/shared-types` 增量新增：

- `IndexStatus`
- `AnswerSessionStatus`
- `AnswerScopeMode`
- `AnswerSession`
- `AnswerCitation`
- `DocumentEvidenceResponse`
- `AnswerRetrievalTraceResponse`
- `OpsAnswerSummaryResponse`

说明：

- 不建议直接复用 `ParseStatus` 承载问答状态。
- `DiagnosisCode` 需要补 `Phase 2A` 的新增枚举。

---

## 9. 不建议的 API 走法

1. 不建议把问答做成 `GET /documents?q=...&retrieval_mode=hybrid`
   原因：会把 list search 和 answer session 混成一类资源。
2. 不建议让前端直接拼 citation
   原因：citation 一旦脱离后端 validator，可信边界立即失效。
3. 不建议把 `retry` 和 `reindex` 合并
   原因：解析失败与问答索引失效是两类不同操作。

---

## 10. Freeze 结论

以下事项已在 [Phase 2A Contract Freeze](/Users/coderlauu/xRag/tech/architecture/2026-04-08-phase-2a-contract-freeze.md) 中固定：

1. `POST /answers` 统一采用 `202 Accepted`
2. `retrieval trace` 在 `P0` 即保留正式资源，不只限内部日志
3. `reindex` 保持独立路径，并对普通用户最小开放
4. `search_result` scope 必须带 snapshot，不允许只传搜索词
5. `POST /documents/{id}/reindex` 的响应结构固定包含 `document_id / job_id / index_status / diagnosis_code`
6. `GET /ops/answer-summary` 在 `P0` 作为正式读取资源存在

以下内容仍属于页面或实现细节，不构成 API contract blocker：

1. `GET /answers/{id}` 的前端轮询节奏
2. `GET /answers/{id}/retrieval` 在首版页面中默认展开还是折叠
