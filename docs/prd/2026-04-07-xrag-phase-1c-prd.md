# xRag Phase 1C PRD

**日期：** 2026-04-07  
**版本：** `v3 / Phase 1C`  
**状态：** active

---

## 1. 一句话定位

在 `Phase 1B` 已稳定的导入与解析基线上，扩展扫描件与链接内容接入能力，并提升检索质量与处理过程可观测性。

---

## 2. 背景

`Phase 1A` 解决了最小导入、搜索、详情闭环。`Phase 1B` 进一步解决了：

- 文本型 PDF 真实解析
- multipart 大文件上传
- 上传与解析失败诊断
- production 部署、观测与热修基线

当前新的核心矛盾已经变化为：

- 扫描版 PDF 仍然无法处理
- 链接类内容仍未进入正式导入闭环
- 搜索结果虽然可用，但相关性和结果解释还不够强
- 用户与运维视角对“文档处理过程”仍缺少更完整的时间线和证据

---

## 3. 本期目标

- 支持扫描版 PDF 的 OCR 解析闭环
- 支持链接正文抓取与入库闭环
- 提升搜索结果排序、匹配解释和结果可读性
- 提供文档级处理时间线与更强的诊断可见性

---

## 4. 非目标

- AI 问答
- 向量检索 / 语义召回
- 浏览器插件
- 团队协作 / 权限模型
- 移动端
- 全自动生产修复与自动发版

---

## 5. 核心场景

### 场景 1：上传扫描版 PDF

用户上传图片型或扫描件 PDF，希望系统能通过 OCR 提取正文，而不是直接失败。

### 场景 2：添加网页链接

用户输入一个正文页面链接，希望系统可以抓取主内容、标题和来源信息，并进入搜索闭环。

### 场景 3：搜索更快定位有效内容

用户通过关键词检索时，希望更容易看到为什么命中、匹配来自标题还是正文，以及不同来源内容的排序差异。

### 场景 4：排查处理过程

用户或运维人员需要知道一份文档从上传、解析、OCR、抓取到搜索投影经历了哪些步骤，哪一步失败，下一步该怎么处理。

---

## 6. 本期范围

### P0 必做

1. 扫描版 PDF OCR 管道
2. 链接正文抓取管道
3. 搜索排序与匹配解释增强
4. 文档处理时间线与诊断展示增强
5. 对应前后端、worker、测试与运维闭环

### P1 预留

1. OCR 多语言支持
2. 链接抓取反反爬与动态页面渲染
3. 搜索高级过滤器
4. 更大范围的 CI 自动修复能力

---

## 7. 验收标准

1. 扫描版 PDF 上传后，可进入 OCR 并生成可搜索文本，且失败时有明确诊断。
2. 链接导入后，可提取标题、正文摘要和来源 URL，并进入搜索与详情页。
3. 搜索结果可以展示匹配解释或结果增强信息，排序质量优于当前基线。
4. 详情页或运维页可以看到文档处理时间线和关键步骤状态。
5. 相关单元、集成、E2E、smoke 覆盖新的接入链路。

---
