# v1 Scope And Tradeoffs

**日期：** 2026-03-31  
**版本：** `v1 / Phase 1A`  
**状态：** 已决定  
**对应原型：** [prototype/v1](/Users/coderlauu/xRag/prototype/v1/index.html)

## 1. 本文档目的

这份文档用于回答一个具体问题：

`v1 这次到底交付了什么，没有交付什么，以及为什么这么取舍。`

它是 PRD 与技术方案之间的边界文档，后续 coding 阶段应优先参考这里的范围定义。

---

## 2. v1 实际交付范围

### 已纳入本次原型

- 导入页 / 搜索页 / 详情页三页闭环
- 手动文本保存
- 文件上传
- 导入状态展示
- 搜索结果与详情联动
- 标签展示与基础筛选
- 空结果态与失败态
- 基于本地状态的 mock 数据驱动

### 原型层模拟但非真实实现

- PDF 解析成功 / 失败的异步状态流
- 文档搜索索引
- 处理队列自动刷新
- 详情页标签编辑写回

这些能力在 `v1` 中用于演示业务流程，不代表已经有真实后端或正式数据链路。

---

## 3. 本次刻意不做的内容

### 不做，是为了控制范围

- 语义搜索
- AI 问答
- 真实 OCR
- 链接正文抓取
- 浏览器插件
- 排序增强
- 多端同步
- 团队协作

### 原因

- Phase 1A 的目标是验证最小闭环，而不是堆叠能力。
- 若在 `v1` 同时引入 AI、OCR、插件，会让技术方案过早复杂化。
- 先跑通用户是否愿意导入、是否能找回，比更高级能力更关键。

---

## 4. 关键取舍

### 4.1 先做关键词搜索，不做语义检索

**取舍：**
优先实现可理解、可验证、可演示的关键词搜索结果。

**原因：**
这能更快验证“找回”是否成立，也更利于技术方案分阶段设计。

### 4.2 用静态原型站而不是正式前端工程

**取舍：**
当前阶段使用 `prototype/v1` 的静态 HTML/CSS/JS 结构。

**原因：**

- 可以更快表达交互
- 不引入工程化复杂度
- 更适合版本化存档与对比

### 4.3 保留 mock data 层

**取舍：**
在 `mock-data/` 和 `store.js` 中明确保留种子数据和本地状态模拟。

**原因：**

- 为产品评审和技术方案拆解提供可控样本
- 让后续真实接口接入时有明确替换点

### 4.4 状态流先做强反馈

**取舍：**
即使是原型，也必须展示 `pending / processing / success / failed`。

**原因：**

- 这是产品价值的一部分
- 也是未来数据模型和异步任务设计的重要输入

## 5. 与现有 ADR 的关系

仓库中已有 [ADR-001-tags-design](/Users/coderlauu/xRag/docs/decisions/ADR-001-tags-design.md)，其方向是：

- 从 free-text 标签输入转为预定义标签选择
- 支持新增标签入口

当前 `v1` 原型尚未完全落地该 ADR，仍保留自由输入式标签编辑。

因此后续技术方案和 `v2` 迭代要注意：

- `ADR-001` 代表标签系统的下一步设计方向
- 但它不是当前 `v1` 已实现事实

---

## 6. 技术方案阶段必须承接的事项

后续技术方案必须显式回答以下问题：

1. 文档对象如何持久化
2. 搜索索引如何建立与更新
3. `processing / failed` 状态由什么任务机制驱动
4. 文件解析与关键词搜索如何解耦
5. 标签体系是否按 `ADR-001` 收敛为可管理标签库

---

## 7. 下一版本建议

### 优先方向

- 让标签系统与 ADR-001 对齐
- 明确真实数据模型与 API 边界
- 决定链接保存是进入 `v2` 还是继续延后

### 不建议立刻进入的方向

- 直接上 AI 问答
- 直接把原型改成完整前端工程
- 在未稳定对象模型前引入复杂搜索能力
