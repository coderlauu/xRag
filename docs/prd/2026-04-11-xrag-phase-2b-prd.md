## 1. 一句话定位

在 `Phase 2A` 已完成可信问答闭环后，把问答体验从“能答”升级到“能控、能查、能回看”，让用户既能精确限制范围，也能理解答案与检索链路。

---

## 2. 背景

`Phase 2A` 已经证明三件事成立：

- 用户可以直接对个人知识库发起问题
- 系统可以基于引用、拒答、freshness 和评估门槛建立可信回答
- 工程主链路已具备 release-ready 的部署、回滚和生产排查基线

但当前产品仍有明显缺口：

- 范围控制仍停留在 `全库 / 当前搜索结果 / 单篇文档`，不足以支持 power-user 的真实筛选方式
- 用户虽然能看到 citation，但还不够容易理解“为什么是这些证据、为什么不是其他证据”
- 检索 trace、未入答原因和 URL state 已有设计雏形，但还没有成为清晰的产品事实
- 用户完成一次问答后，缺少“回看最近问题”和“继续提问”的稳定入口
- `ops` 和评估指标已有最小 gate，但没有形成可持续使用的产品面板

因此 `Phase 2B` 的核心不是再发明新的 AI 能力，而是把 `Phase 2A` 已明确下放的 `P1` 项重新组合，冻结下一轮增强的真实边界。

---

## 3. 本期目标

- 把问答范围扩展到 `标签 / 来源类型 / 时间范围`
- 把检索解释、未入答原因和 URL state 产品化，形成可复现的 retrieval workbench
- 把证据展示从“平铺 citation”升级到“按 claim 分组、可读、可核查”的 evidence package
- 提供最近问题、回看旧答案和继续提问入口，但不扩张为多轮 assistant
- 在不破坏现有评估门槛的前提下，定义 `ops` 面板增强的产品边界与优先级

---

## 4. 非目标

- 开放互联网联网回答
- 多轮 assistant memory、隐式上下文累积或自动扩 scope
- 单文档摘要、多文档对比摘要、相关内容推荐
- 搜索高级过滤器重构
- 多模型路由与更细粒度模型策略
- 团队协作 / 权限模型
- 浏览器插件
- 移动端

---

## 5. 核心场景

### 场景 1：在明确范围内提问

用户想只针对某个标签、某类来源或某段时间内的资料提问，希望系统严格限定在该范围，不把其他内容混入答案。

### 场景 2：理解为什么这些证据进入答案

用户拿到答案后，希望理解每条 claim 由哪些证据支持，freshness 如何影响可信度，以及有哪些结果被召回但没有进入答案。

### 场景 3：复现一次检索链路

用户、产品或工程希望在检索实验台里复现一次问答对应的 retrieval path，看到 lexical / semantic / rerank 各自贡献了什么。

### 场景 4：继续一条已经验证过的问答

用户做完第一次问答后，希望从最近问题中回到旧结果，或者在同一明确 scope 下继续追问，而不是从空白状态重新开始。

### 场景 5：判断这轮增强是否仍然可信

产品、工程和运维希望知道 `Phase 2B` 的增强没有破坏 `citation / refusal / freshness / release-readiness` 这四条既有边界。

---

## 6. 优先级原则

1. 先增强“范围控制与解释力”，再增强“辅助型运维可视化”
2. 先让用户明确知道“当前问的是哪一组资料”，再补更复杂的体验功能
3. 解释页面、答案页面、详情页和历史入口必须共享同一套问答事实源
4. `继续提问` 只允许建立在显式 scope 和显式上一条 session 之上，不得演化成隐式 memory
5. 如果任何增强会威胁 `citation / refusal / freshness / release-readiness`，优先降 scope，而不是继续扩功能

---

## 7. 需求细化与优先级拆分

### 7.1 P0 核心交付

| ID | 能力 | 需求细化 | 为什么是 P0 |
| --- | --- | --- | --- |
| `P0-01` | 扩展范围控制 | 在现有 `全库 / 当前搜索结果 / 单篇文档` 之上，补齐 `标签 / 来源类型 / 时间范围` 三类显式 scope。 | 这是 `Phase 2B` 最直接的用户价值，也是后续解释力成立的前提。 |
| `P0-02` | 检索实验台基线 | 把 lexical / semantic / rerank 贡献、未入答原因和 URL state 变成正式产品界面。 | 若用户无法理解“为什么这些文档进答”，解释力和排查能力都不完整。 |
| `P0-03` | 证据包增强 | 按 claim 组织答案证据，支持多引用组合，并强化 freshness 提示。 | `Phase 2A` 已证明“有 citation”，`Phase 2B` 需要证明“citation 更可读、更可核查”。 |
| `P0-04` | 问题历史与继续提问 | 提供最近问题列表、旧答案回看和继续提问入口，但继续提问仍创建新的显式 session。 | 这是把可信问答真正变成日常工作流的最低门槛。 |

### 7.2 P1 补强

| ID | 能力 | 需求细化 | 为什么下放到 P1 |
| --- | --- | --- | --- |
| `P1-01` | 评估与运维面板增强 | 在 `ops` 板上系统展示 recall、groundedness、citation coverage、latency、cost、backlog 等趋势，以及最近失败会话聚类。 | 这对内部调优价值很高，但不是最先决定用户体验升级是否成立的路径。 |

### 7.3 明确延后

以下能力保留到 `Phase 2B+` 或后续版本，不放进本轮实现范围：

| 能力 | 原因 |
| --- | --- |
| 单文档摘要、多文档对比摘要 | 它们会再次把产品重心拉向“生成更多内容”，而不是“先把可信边界和解释力做扎实”。 |
| 相关内容推荐 | 属于消费增强，不是 `Phase 2B` 的主线。 |
| 搜索高级过滤器增强 | 容易与范围控制、URL state 和 search page 重构耦合，当前不应一起引入。 |
| 多模型路由与更细粒度模型策略 | 当前优先级仍然是产品边界与解释力，而不是模型调度平台。 |
| 多轮 assistant memory / 自动扩 scope | 会直接改变 trust model，不适合在本轮默认带入。 |
| 联网问答、浏览器插件、团队协作、移动端 | 与当前单用户私有知识库主线不一致。 |

先交付：

- `P0-01` 扩展范围控制
- `P0-03` 证据包增强

目标：

- 先让用户明确“问的范围”与“答案依据”同时变得更可控、更可读。

### Slice B：检索解释与历史工作流

继续补齐：

- `P0-02` 检索实验台基线
- `P0-04` 问题历史与继续提问

目标：

- 让用户与工程都能回放一条问答链路，并把单次问答升级为可复用的工作流入口。

### Slice C：运维与评估面板增强

最后再补：

- `P1-01` 评估与运维面板增强

目标：

- 在不打断主问答心智的前提下，把内部诊断和版本守护能力产品化。

## 9. 验收标准

### 9.1 Phase 2B Gate

1. 用户可在提问前或提问时明确限定 `标签 / 来源类型 / 时间范围`，且当前 scope 在问答结果中清晰可见。
2. 检索实验台能复现当前问答对应的 retrieval path，并明确展示哪些结果被召回但未进入答案，以及原因是什么。
3. 每个 `answered` 结果都能按 claim 展示支持证据，freshness 风险不会被埋在细节里。
4. 最近问题列表允许用户回看旧答案，`继续提问` 会创建新的 session，而不是把旧答案偷偷改写成新的结果。
5. `citation / refusal / freshness / release-readiness` 四条既有边界不得回退。

### 9.2 Phase 2B 完整度补强

1. `ops` 板中的趋势指标与 [Phase 2A Evaluation Plan](/Users/coderlauu/xRag/docs/process/2026-04-07-phase-2a-evaluation-plan.md) 保持完全一致。
2. 产品、工程和运维可以通过页面快速定位失败集中在 `scope / retrieval / evidence / provider / freshness` 哪一层。
3. `Phase 2B` 的增强不会让用户误以为系统已经支持联网问答、多轮记忆或更宽泛的自动代理能力。

---

## 10. 对技术方案的要求

进入技术方案评估前，必须补齐或回答：

- `Phase 2B` 的交互 delta，明确哪些页面是主入口、哪些页面是诊断入口
- 扩展 scope 的 typed contract 与可见性规则
- retrieval explain / excluded reason / evidence grouping 的共享事实源
- 问题历史、继续提问与旧 session 的边界关系
- `ops` 面板的指标来源、更新节奏与角色定位

在这些边界没冻结前，不进入新的实现 freeze。
