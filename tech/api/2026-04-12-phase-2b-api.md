## 1. 设计目标

`Phase 2B` 的 API 目标不是重做 `answers`，而是在现有 `Phase 2A` contract 上增量补齐：

1. typed scope filters
2. recent answers / continue asking
3. evidence groups
4. retrieval workbench summary
5. `ops` 增强的可扩边界

基础前缀继续保持：

```text
/api/v1
```

---

## 2. API 设计原则

1. 现有 `POST /answers`、`GET /answers/{id}`、`GET /answers/{id}/retrieval` 保持路径稳定
2. 新能力优先通过 additive fields 或 collection resource 进入，不重命名既有字段
3. `scope`、`history`、`evidence` 与 `retrieval` 必须共享同一套 session 事实源
4. 不把 URL state、页面临时状态或 explain rerun 设计成 API contract
5. `ops` 如果继续增强，也必须沿用现有 `/ops/*` 资源族

---

## 3. 现有接口的保留与扩展方向

### 3.1 `GET /api/v1/documents`

继续作为：

- keyword search 基线
- 文档列表与搜索页事实源
- typed scope filters 的词汇来源

结论：

- `ListDocumentsQuery` 里的 `source_type / tags / date_from / date_to / index_status` 继续保留
- `Phase 2B` 的 scope filters 应尽量与这些字段保持一致命名

### 3.2 `GET /api/v1/documents/{documentId}` / `evidence` / `reindex`

继续保留现有路径。

结论：

- evidence package 的增强不应把 document evidence 端点重新设计成问答专用资源
- document detail / evidence 仍是 citation 回跳的最终落点

## 4. 推荐的 `answers` contract 增量

### 4.1 `CreateAnswerRequest`

当前请求体只有：

- `question`
- `scope`

推荐增量为：

```json
{
  "question": "只看 2026 年以来的 pdf 和周报，近期哪些事项最值得继续追踪？",
  "scope": {
    "mode": "global",
    "payload": {
      "filters": {
        "tags": ["weekly", "roadmap"],
        "source_types": ["pdf"],
        "date_from": "2026-01-01",
        "date_to": "2026-04-12"
      }
    }
  },
  "continued_from_session_id": "ans_prev_123"
}
```

推荐结论：

- `scope.mode` 保持：
  - `global`
  - `document`
  - `search_result`
- `continued_from_session_id` 作为顶层可选字段新增
- 不新增单独的 `/follow-up` endpoint

### 4.2 `AnswerScope` payload shape

推荐冻结如下：

| `scope.mode` | payload |
| --- | --- |
| `global` | `{ filters?: ScopeFilterSet | null }` |
| `document` | `{ document_id: string }` |
| `search_result` | `{ document_ids: string[]; truncated: boolean; query?: string | null; filters?: ScopeFilterSet | null }` |

其中 `ScopeFilterSet` 为：

```json
{
  "tags": ["weekly", "roadmap"],
  "source_types": ["pdf", "link"],
  "date_from": "2026-01-01",
  "date_to": "2026-04-12"
}
```

当前不建议：

- 新增 `tag`、`source_type`、`time_range` 等 mode
- 把 filters 移到顶层，与 `scope` 平行存在

### 4.3 `GET /api/v1/answers`

推荐新增 collection resource：

```text
GET /api/v1/answers
```

用途：

- recent answers 列表
- history reopen 入口

推荐响应字段：

- `session_id`
- `question`
- `status`
- `scope`
- `scope_summary`
- `continued_from_session_id`
- `created_at`
- `finished_at`

推荐分页方式：

- cursor-based

原因：

- recent history 只需要“可回看、可继续”，不需要完整 thread API

### 4.4 `GET /api/v1/answers/{sessionId}`

现有响应保持不删字段，推荐新增：

- `scope_summary`
- `continued_from_session_id`
- `evidence_groups`

推荐响应片段：

```json
{
  "session_id": "ans_123",
  "status": "answered",
  "scope_summary": "标签 weekly、roadmap；来源 pdf；时间 2026-01-01 至 2026-04-12",
  "continued_from_session_id": "ans_prev_001",
  "answer_summary": "基于当前范围，近期优先级最高的是检索 explain、范围控制和 evidence package。",
  "citations": [],
  "evidence_groups": [
    {
      "claim_slot": "claim_1",
      "claim_text": "近期优先级最高的是检索 explain、范围控制和 evidence package。",
      "citations": []
    }
  ]
}
```

兼容要求：

- flat `citations` 继续保留
- `evidence_groups` 是 additive，不替换旧字段

### 4.5 `GET /api/v1/answers/{sessionId}/retrieval`

现有资源继续保留，但响应推荐从“只有 items”升级为“summary + items”：

推荐新增 summary 字段：

- `query_normalized`
- `eligible_document_count`
- `lexical_hit_count`
- `semantic_hit_count`
- `merged_hit_count`
- `rerank_strategy`
- `latency_ms`

推荐把 `exclusion_reason` 冻结为共享 enum：

- `deduplicated`
- `rerank_cutoff`
- `answer_budget`
- `low_support`
- `citation_unready`

结论：

- retrieval workbench 不需要新 endpoint
- URL state 继续由 web 自行维护

## 6. `ops` API 的冻结边界

`Phase 2B` 当前不建议新增新的 `/ops` 一级资源。

推荐方向：

- 若 `P1-01` 进入实现，优先扩展现有：
  - `GET /api/v1/ops/answer-summary`
  - `GET /api/v1/ops/incidents`
  - `GET /api/v1/ops/deployments/latest`

原则：

1. 指标口径必须与 [Phase 2A Evaluation Plan](/Users/coderlauu/xRag/docs/process/2026-04-07-phase-2a-evaluation-plan.md) 同源
2. 不新增第二套 trends API 再去重复造数

---

## 7. 兼容性要求

1. 旧客户端继续可以用当前 `question + scope` 请求体创建 answer session
2. 旧 session 继续可以通过 `GET /answers/{id}` 正常读取
3. 对没有 claim 数据的旧 session，允许返回空 `evidence_groups`
4. `search_result` 继续要求 `document_ids + truncated`
5. 新字段必须同步进入：
   - OpenAPI
   - `packages/shared-types`
   - `packages/api-client`

---

## 8. 明确不建议的 API 方向

当前不建议新增：

- `POST /api/v1/answers/{id}/continue`
- `POST /api/v1/answers/{id}/rerun-retrieval`
- 独立的 `/api/v1/retrieval-workbench/*`
- 直接暴露 prompt、chain-of-thought 或模型中间推理轨迹

原因：

- 这些路径要么破坏资源模型，要么会把产品边界扩到当前版本不打算承诺的区域
