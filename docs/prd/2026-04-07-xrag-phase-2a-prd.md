# xRag Phase 2A PRD

**日期：** 2026-04-07  
**版本：** `v4 / Phase 2A`  
**状态：** active

---

## 1. 一句话定位

在已完成统一导入、检索与可观测性基线上，引入可引用的 AI 问答与混合检索，让资料从“可找回”升级到“可直接复用”。

---

## 2. 背景

`Phase 1A` 到 `Phase 1C` 已经连续验证了三件事：

- 内容可以被统一导入
- 内容可以被稳定搜索找回
- 处理过程、失败诊断与生产观测已有最小可用基线

当前新的核心矛盾不再是“资料进不来”或“资料搜不到”，而是：

- 用户仍然需要自己打开多篇文档、手动比对和总结
- 关键词搜索对“换一种说法提问”的支持仍然有限
- 即使搜索命中，用户也缺少一个可直接消费的答案层
- 如果直接引入 AI，但没有证据链与拒答机制，产品信任会明显下降

---

## 3. 本期目标

- 支持基于个人知识库的 `AI 问答`
- 支持 `keyword + semantic` 混合检索
- 支持基于搜索结果、标签、来源类型、时间范围、单篇文档的问答范围控制
- 提供可引用、可跳转、可审计的答案证据链
- 建立导入内容到向量索引的更新、诊断与回补基线
- 建立答案质量、检索质量、延迟与成本的最小评估框架

---

## 4. 非目标

- 开放互联网联网回答
- 无证据强答
- 团队协作 / 权限模型
- 浏览器插件
- 移动端
- 全自动 agent 工作流
- 更大范围的动态页面抓取和反反爬能力

---

## 5. 核心场景

### 场景 1：跨文档提问并获得带引用答案

用户输入一个自然语言问题，希望系统不仅返回搜索结果，还能直接给出答案，并标明答案来自哪些文档和片段。

### 场景 2：限定范围提问

用户已经筛出一组搜索结果，或只想问某个标签、来源类型、时间段内的内容，希望问答严格限定在该范围内。

### 场景 3：证据不足时拒答

用户提出的问题在知识库中证据不足时，系统应明确说明“当前证据不足”，而不是生成似是而非的内容。

### 场景 4：追溯答案来源

用户看到答案后，希望快速查看引用片段、跳到原始文档详情，并理解为什么这些证据被选中。

### 场景 5：新导入内容快速进入可问答状态

用户刚上传文件、OCR 文档或导入链接后，希望内容能在合理时间内进入 AI 问答索引；若失败，需要有诊断和修复入口。

---

## 6. 本期范围

### P0 必做

1. 基于导入知识库的 `AI 问答`
2. `keyword + semantic` 混合检索
3. 问答范围控制
4. 答案证据链
   - 引用文档
   - 引用片段
   - 跳回详情页
   - 无证据拒答
5. 导入内容到向量索引的更新、失败诊断与回补
6. 检索 / 问答质量、延迟、成本的最小评估与运维基线

### P1 预留

1. 单文档摘要、多文档对比摘要
2. 相关内容推荐
3. 问答历史与保存会话
4. 搜索高级过滤器增强
5. 多模型路由与更细粒度的模型策略

---

## 7. 验收标准

1. 用户可输入自然语言问题，并获得基于已导入文档生成的答案。
2. 每个答案都必须附带明确引用，用户可跳转到对应文档详情页验证来源。
3. 语义改写后的问题，相比纯关键词基线，召回质量有可感知提升。
4. 当知识库证据不足时，系统能稳定给出拒答或提示补充范围，而不是无依据回答。
5. 用户可以限定问答范围到搜索结果、标签、来源类型、时间范围或单篇文档。
6. 新导入的文本、PDF、OCR、链接内容能够在可接受的 freshness 窗口内进入问答索引；失败时可见诊断和回补入口。
7. 本期至少形成一套可重复执行的评估数据集与质量基线，用于衡量检索质量、答案 groundedness、延迟和成本。

---

## 8. 对技术方案的要求

进入实现前，必须补齐：

- `Phase 2A` 的交互说明与原型
- 混合检索与向量索引架构
- chunk / embedding / retrieval / rerank 的最小策略
- 问答会话、答案引用、检索轨迹与索引任务的数据模型
- 问答 API、引用 contract 与拒答 contract
- 导入到索引的异步状态流与诊断方案
- 质量评估、灰度发布、feature flag 与回滚方案
