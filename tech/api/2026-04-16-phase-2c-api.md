## 1. 本文档目的

这份文档用于冻结 `Phase 2C` 的 API 方向：

- 哪些现有 `/ops/*` 资源继续保留
- 是否新增面向治理主板的 read model 资源
- 如何区分 runtime quality 和 evaluation quality

本阶段不直接生成 OpenAPI，只先冻结资源族方向和响应分层。

---

## 2. 当前 API 事实

当前已有的 `ops` 资源：

- `GET /api/v1/ops/health-summary`
- `GET /api/v1/ops/incidents`
- `GET /api/v1/ops/answer-summary`
- `GET /api/v1/ops/deployments/latest`

当前问题：

1. `answer-summary` 混合 readiness 与 runtime quality，但没有趋势、evaluation 指标和正式 blocking reason。
2. `incidents` 只有平铺列表，没有 clusters 和推荐动作。
3. `deployments/latest` 只有 latest 视角，没有 release guard 所需的历史关联。
4. 当前页面自己计算 readiness rate、状态文案和 recommended actions，API 没有正式表达这些判断。

---

## 3. 资源方向与结论

### 3.1 现有资源继续保留

以下路径保持兼容：

- `GET /api/v1/ops/health-summary`
- `GET /api/v1/ops/incidents`
- `GET /api/v1/ops/answer-summary`
- `GET /api/v1/ops/deployments/latest`

原因：

- 旧页面和现有 SDK 仍依赖它们
- 它们仍适合作为 drill-down 或细节读取面

### 3.2 新增治理 read model

`Phase 2C` 推荐新增两个主资源：

1. `GET /api/v1/ops/overview`
2. `GET /api/v1/ops/trends`

**推荐理由**

- `overview` 负责表达“现在能不能答、哪里最危险、先处理什么”
- `trends` 负责表达固定时间窗口下的质量和 readiness 漂移
- 避免继续把主板逻辑塞在前端或堆进 `answer-summary`

### 3.3 当前不新增 `/api/v1/evals/*`

`Phase 2C / P0` 不推荐新增：

- `GET /api/v1/evals/runs`
- `GET /api/v1/evals/cases`
- `POST /api/v1/evals/*`

原因：

- 当前目标是把治理结果产品化，不是把评估管理系统直接做进产品
- evaluation runs 可先作为 `ops` read model 的内部事实源

## 4. 推荐资源形状

### 4.1 `GET /api/v1/ops/overview`

#### 目标

返回治理主板首屏所需的聚合块。

#### 推荐响应层级

```json
{
  "generated_at": "2026-04-16T00:00:00Z",
  "readiness": {},
  "runtime_quality": {},
  "evaluation_quality": {},
  "incident_summary": {},
  "release_guard": {},
  "recommended_actions": []
}
```

#### 推荐块语义

- `readiness`
  - 来自 `documents`
  - 至少包含：
    - `queued_count`
    - `chunking_count`
    - `embedding_count`
    - `ready_count`
    - `stale_count`
    - `failed_count`
    - `readiness_rate`
    - `freshness_lag_p95_ms`
    - `blocking_reason`
- `runtime_quality`
  - 来自 `answer_sessions + answer_citations`
  - 至少包含：
    - `latency_p50_ms`
    - `latency_p95_ms`
    - `avg_token_cost_usd`
    - `citation_coverage`
    - `refusal_rate`
- `evaluation_quality`
  - 来自 `evaluation_runs`
  - 至少包含：
    - `latest_run_ref`
    - `latest_completed_at`
    - `recall_at_10`
    - `mrr`
    - `hit_in_answer_rate`
    - `groundedness`
    - `citation_coverage`
    - `refusal_precision`
- `incident_summary`
  - 聚类摘要而不是平铺列表
- `release_guard`
  - 以 `deployment_records + latest deployment + health` 形成风险判断
- `recommended_actions`
  - 返回结构化动作，而不是只返回散文文案

### 4.2 `GET /api/v1/ops/trends`

#### 目标

返回固定时间窗口下的趋势序列，供治理主板画图。

#### 推荐查询参数

- `window`
  - `24h | 7d | 30d`

#### 推荐响应层级

```json
{
  "window": "7d",
  "generated_at": "2026-04-16T00:00:00Z",
  "series": []
}
```

#### 推荐序列语义

每条序列至少包含：

- `metric`
- `source`
  - `runtime` 或 `evaluation`
- `granularity`
- `points`
  - `{ ts, value }[]`

#### 推荐首批 metric

- runtime
  - `readiness_rate`
  - `embedding_backlog`
  - `freshness_lag_p95_ms`
  - `citation_coverage`
  - `latency_p95_ms`
  - `avg_token_cost_usd`
- evaluation
  - `groundedness`
  - `refusal_precision`
  - `recall_at_10`

### 4.3 `GET /api/v1/ops/incidents`

当前路径继续保留为列表读面。

#### 方向

- 可以做 additive 扩展，例如增加：
  - `occurred_at`
  - `related_surface`
  - `action_code`
- 但不建议把 clusters 直接塞回此列表响应，避免混合语义

### 4.4 `GET /api/v1/ops/answer-summary`

当前路径继续保留，但角色收窄为：

- 兼容旧页面
- 提供最小 runtime readiness + quality summary

#### 关键约束

1. `refusal_rate` 保持原字段名和原语义
2. 不在该响应里硬塞 `groundedness` 或 `refusal_precision`
3. 若新增字段，只做 additive，不改旧字段语义

### 4.5 `GET /api/v1/ops/deployments/latest`

当前路径继续保留 latest 读面。

#### 关键约束

- latest deployment 仍可直接读取
- release guard 的历史对比不再只靠这个接口承担

## 5. Shared Types 方向

后续 contract freeze 需要新增的 shared types 至少包括：

- `OpsOverviewResponse`
- `OpsReadinessSnapshot`
- `OpsRuntimeQualitySummary`
- `OpsEvaluationQualitySummary`
- `OpsIncidentCluster`
- `OpsReleaseGuard`
- `OpsRecommendedAction`
- `OpsTrendsResponse`
- `OpsTrendSeries`

现有 shared types 继续保留：

- `OpsAnswerSummaryResponse`
- `OpsHealthSummaryResponse`
- `OpsIncidentListResponse`
- `LatestDeploymentResponse`

---

## 6. 兼容性要求

1. 旧 `ops` 页面可以在一段时间内继续使用现有四个接口。
2. 新 `ops` 主板在 `Phase 2C` 实现期迁移到 `overview + trends`。
3. 现有 `packages/shared-types`、`packages/api-client` 和 OpenAPI 必须在 contract freeze 之后同步更新。
4. 不得把 `runtime quality` 和 `evaluation quality` 混到同一个旧 DTO 里。

---

## 7. 明确不建议的 API 方向

当前不建议新增：

- `/api/v1/ops/dashboard-snapshots`
- `/api/v1/ops/auto-remediation/*`
- `/api/v1/evals/*`
- 任何会把 `Ask / Search / Detail` 变成治理主入口的资源

原因：

- 会扩大版本范围
- 会让 contract freeze 牵扯执行编排和评测管理平台

---

## 8. Freeze-Ready 结论

1. `Phase 2C` 推荐新增 `overview + trends` 两个治理 read model。
2. 现有 `answer-summary / incidents / deployments/latest / health-summary` 保持兼容和细节读面角色。
3. `refusal_rate` 与 `refusal_precision` 必须正式分离。
4. 不新增 `evals` 资源族，评估结果先通过 `ops` read model 暴露。
5. 在这些结论下，API 面没有新的硬 blocker，可以进入 contract freeze。
