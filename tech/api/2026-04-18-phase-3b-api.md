## 1. 本文档目的

这份文档用于冻结 `Phase 3B` 的 API 方向：

- 哪些现有 ops diagnostic 路径继续保留
- recovery candidate / action / audit / rollback plan 应如何挂在 `ops` 资源族下
- 现有 document retry/reindex API 与新 recovery API 如何分工
- 哪些响应应复用既有 DTO，哪些必须形成新的 action contract

本阶段不直接生成 OpenAPI，只先冻结资源方向和响应分层。

---

## 2. 当前 API 事实

当前已存在的相关资源：

- `GET /api/v1/ops/samples`
- `GET /api/v1/ops/replays/answer-sessions/:sessionId`
- `GET /api/v1/ops/replays/documents/:documentId`
- `GET /api/v1/ops/deployments/compare`
- `POST /api/v1/documents/:documentId/retry`
- `POST /api/v1/documents/:documentId/reindex`
- `GET /api/v1/jobs/:jobId`
- `GET /api/v1/answers/:sessionId`
- `GET /api/v1/answers/:sessionId/retrieval`

当前问题：

1. 现有 diagnostic paths 都是 read-only。
2. 现有 document retry/reindex 是 direct mutation，不具备 Phase 3B 所需的 candidate、preview、operator reason、action id 和 audit。
3. 现有 job status 只能表达 document job，不能表达跨对象 recovery action。

---

## 3. 资源方向与结论

### 3.1 现有资源继续保留

以下路径继续保留，不重命名、不改语义：

- `GET /api/v1/ops/samples`
- `GET /api/v1/ops/replays/answer-sessions/:sessionId`
- `GET /api/v1/ops/replays/documents/:documentId`
- `GET /api/v1/ops/deployments/compare`
- `POST /api/v1/documents/:documentId/retry`
- `POST /api/v1/documents/:documentId/reindex`
- `GET /api/v1/jobs/:jobId`

原因：

- v7 diagnostic workflow 已依赖这些 read resources。
- Document retry/reindex 作为既有 direct API 保留兼容，但不作为 `/ops` recovery UI 的主路径。

### 3.2 `Phase 3B` 推荐新增的 `ops` recovery 资源

推荐新增五类资源：

1. `GET /api/v1/ops/recovery/candidates`
2. `POST /api/v1/ops/recovery/actions/preview`
3. `POST /api/v1/ops/recovery/actions`
4. `GET /api/v1/ops/recovery/actions/:actionId`
5. `GET /api/v1/ops/recovery/actions/:actionId/audit`
6. `GET /api/v1/ops/recovery/rollback-plan`

**推荐理由**

- `candidates`
  - 负责把 v7 diagnostic facts 转成可审查下一步
- `actions/preview`
  - 负责 dry-run / preview，不创建 execution side effect
- `actions`
  - 负责人工确认后的 execution envelope
- `actions/:actionId`
  - 负责 action status polling
- `actions/:actionId/audit`
  - 负责 before/after facts 和 status timeline
- `rollback-plan`
  - 负责从 deployment compare 生成 manual checklist

### 3.3 当前不新增的路径

`Phase 3B / P0` 不推荐新增：

- `POST /api/v1/ops/recovery/rollback`
- `POST /api/v1/ops/remediate`
- `POST /api/v1/ops/recovery/actions/:actionId/retry`
- `POST /api/v1/answers/:sessionId/rerun`
- `/api/v1/ops/logs/*`

原因：

- 当前目标是受控人工 recovery，不是自动化处置平台。
- Answer rerun 若直接落到 answers 资源下，很容易被误解为重答并替换用户可见答案。

## 4. 推荐资源形状

### 4.1 `GET /api/v1/ops/recovery/candidates`

#### 目标

返回从 diagnostic sample、replay 或 deployment compare 派生的 recovery candidate。

#### 推荐查询参数

- `source_type`
  - `diagnostic_sample | answer_session_replay | document_replay | deployment_compare`
- `source_ref`
  - 具体 source id 或 sample id
- `action_type`
  - 可选
- `risk_level`
  - 可选

#### 每个 item 至少包含

- `candidate_id`
- `source_type`
- `source_ref`
- `action_type`
- `target_type`
- `target_refs`
- `risk_level`
- `preconditions`
- `recommendation_state`
- `preview_ref`

### 4.2 `POST /api/v1/ops/recovery/actions/preview`

#### 目标

对候选 action 做 preview/dry-run，不执行 mutation。

#### 推荐请求

```json
{
  "candidate_id": "document_reindex:...",
  "action_type": "document_reindex",
  "target_type": "document",
  "target_refs": ["..."]
}
```

#### 推荐响应

- `preview_id`
- `candidate_id`
- `action_type`
- `target_type`
- `target_refs`
- `risk_level`
- `preconditions`
- `blocked_reason`
- `estimated_blast_radius`
- `idempotency_key`

### 4.3 `POST /api/v1/ops/recovery/actions`

#### 目标

人工确认后创建并执行 recovery action。

#### 推荐请求

```json
{
  "candidate_id": "document_reindex:...",
  "preview_id": "...",
  "idempotency_key": "...",
  "reason": "Reindex failed document after operator review."
}
```

#### 推荐响应

- `action_id`
- `status`
- `action_type`
- `target_type`
- `target_refs`
- `created_at`
- `status_ref`

### 4.4 `GET /api/v1/ops/recovery/actions/:actionId`

#### 目标

返回 action status，供 `/ops` 轮询或刷新。

#### 推荐响应

- `action_id`
- `status`
- `action_type`
- `target_type`
- `target_refs`
- `queue_job_refs`
- `diagnosis_code`
- `error_message`
- `created_at`
- `started_at`
- `completed_at`
- `updated_at`

### 4.5 `GET /api/v1/ops/recovery/actions/:actionId/audit`

#### 目标

返回 action 审计详情。

#### 推荐响应

- `action`
- `source_facts`
- `preview`
- `before_facts`
- `after_facts`
- `status_timeline`
- `manual_follow_up`

### 4.6 `GET /api/v1/ops/recovery/rollback-plan`

#### 目标

基于 deployment compare 返回 manual rollback checklist。

#### 推荐查询参数

- `deployment_record_id`
- `window`

#### 推荐响应

- `generated_at`
- `deployment_record_id`
- `compare_ref`
- `affected_samples`
- `quality_delta_summary`
- `smoke_summary`
- `confidence`
- `missing_evidence`
- `manual_checklist`

## 6. Answer Quality Gate 边界

以下 API 方向不触发 answer-quality behavior change：

- read-only answer diagnostic candidate
- read-only answer replay refresh
- audit 中引用既有 answer facts

以下方向必须先进入 `answer-quality-gate`：

- 重新运行 answer orchestration
- 改写 `answer_sessions.answer_summary`
- 改写 `answer_citations` 或 `answer_claims`
- 改写 freshness/refusal 语义
- 将 rerun 结果暴露给终端用户

---

## 7. Freeze-Ready 结论

1. `Phase 3B` 的 API 增量应挂在 `ops/recovery` 资源族下。
2. 现有 document retry/reindex API 保留，但不作为 `/ops` recovery 主路径。
3. Recovery action API 必须 preview-first，create action 必须携带 idempotency key 和 reason。
4. Action status 与 audit 是新的 SDK/OpenAPI contract surface。
5. Rollback plan 是 `GET` read model，不新增 rollback mutation。
6. 下一步可以进入 contract freeze，锁定路径、query、request/response DTO、状态枚举和值对象形状。
