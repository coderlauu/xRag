# v4 Handoff

**日期：** 2026-04-07  
**版本：** `v4 / Phase 2A`  
**状态：** active

---

## 1. 当前版本一句话目标

在统一导入、检索与可观测性基线上，引入可引用的 AI 问答与混合检索，让资料从可找回走向可直接复用。

---

## 2. 本轮边界

### 必须实现

- 基于个人知识库的 `AI 问答`
- `keyword + semantic` 混合检索
- 问答范围控制
- 答案证据链与无证据拒答
- 导入内容到向量索引的 freshness、诊断与回补基线
- 对应的交互、评估、运维与发布策略

### 明确不做

- 开放互联网联网回答
- 团队协作 / 权限模型
- 浏览器插件
- 移动端
- 全自动 agent 工作流
- 更大范围的动态页面抓取和反反爬能力

### 原型 / 文档中存在但本轮只做预留

- 单文档摘要、多文档对比摘要
- 相关内容推荐
- 问答历史与保存会话
- 搜索高级过滤器增强
- 多模型路由与更细粒度的模型策略

### 当前优先级拆分

- `P0`: 问答入口、混合检索、最小范围控制、证据链、拒答、freshness 可见性、评估门槛
- `P1`: 扩展范围控制、检索实验台增强、评估与运维面板增强、证据包增强、问题历史
- `deferred`: 摘要、推荐、高级过滤器增强、多模型路由及其他非主线能力

## 6. 当前阶段输出要求

- 只围绕 `P0` 推进 implementation lanes，不扩张到 `P1` 和 `deferred`
- 主线程先把 `contract freeze` 落到 `schema / migration / shared-types / OpenAPI / SDK`
- 以 `runtime contracts` 约束数据库基线、provider/env 与多队列命名
- 子 agent 只能接手写入边界清晰的 `api / worker / web / validation` lane
- 若任一 lane 触碰 contract 漂移，立即切回主线程收口

---

## 7. 当前 implementation lanes 的主线程锁定面

1. `schema` 的表、字段、enum、索引与保留策略
2. `shared-types` 的新增类型、增量字段与 diagnosis code
3. `API contract` 的路径、请求/响应、状态码与 terminal 语义
4. `citation / scope / state machine / eval contract` 的稳定边界
5. `OpenAPI / SDK / queue payload` 的代码事实源

---

## 8. 已知风险 / 历史决策

- `v1` 到 `v3` 已完成统一导入、检索、OCR、链接抓取、时间线与运维基线，`v4` 必须增量演进，不重写主链路
- 若没有引用与拒答机制，AI 问答会直接破坏产品信任
- 向量索引 freshness、回补和失败诊断会直接影响“刚导入内容为何问不到”的用户体验
- 模型提供商、成本、延迟与隐私边界需要在技术评估阶段先给出受约束的推荐，而不是继续停留在抽象口号
- 当前已对 `pgvector`、多队列、provider contract、scope snapshot、reindex 与 `202 Accepted` 问答接口做出方向性决策，并已完成 implementation freeze；下一步是在 contract freeze 基线上由主线程先落 `schema / documents / answers / ops / shared-types / OpenAPI / SDK` 的代码事实源，再进入 `worker-runtime / indexing / provider / answer / ask / search-detail / ops / tests` 并行 lane
